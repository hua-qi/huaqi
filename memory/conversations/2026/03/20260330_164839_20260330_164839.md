---

### 🔧 常见补救方案（对号入座）

**情况A：刚合并，还没push（最简单）**
```bash
# 找到合并前的那个commit hash（通常是HEAD~1）
git reset --hard HEAD~1
# 完事儿，世界清净了
```

**情况B：已经push到远程了（feat是你的个人分支）**
```bash
# 方案1：强制回滚（如果只有你一个人在用这个分支）
git reset --hard HEAD~1
git push --force-with-lease

# 方案2：稳妥撤销（保留历史记录，适合已提MR的情况）
git revert -m 1 <合并提交的hash>
# 这会生成一个新的commit来"抵消"合并，历史干净
```

**情况C：最惨的——你在dev分支上执行了合并（污染了主干）**
```bash
# 赶紧revert，别让别人基于错误的dev开发了
git checkout dev
git revert -m 1 <合并提交的hash>
git push
```

---

### 💡 以后防手滑的小技巧

```bash
# 给git上把安全锁：合并前必须确认
git config --global merge.ff false  # 强制生成merge commit，让你多一次检查机会

# 或者用这个口诀：
# "我在哪，就往哪合" —— 站在feat分支上，就是把dev合进feat（这是对的）
# 如果想把feat合进dev，必须站在dev上
```

---

### 🎯 为什么你会遇到这种情况（不是你的错）

因为你用了 rebase，当你执行 `git rebase dev`（或 `git pull --rebase`）时，Git 会把 dev 上所有的提交**逐个重放**到你的 feat 分支上。结果呢？你的 `git log` 突然多了几十甚至上百条别人的提交，看起来就像你把 dev "吞"了一样。

这其实是个**高级坑**——用 merge 的话，至少只有一个合并 commit，一眼就能看出问题；但 rebase 会让错误"隐形扩散"，直到你推送后别人看到"哇这 MR 怎么改了 80 个文件"才发现。

---

### 💡 一个小优化（可选）

既然只有你在操作，且你已经 cherry-pick 到新分支了：

```bash
# 如果你还想保留原来的分支名（比如远程 MR 已经关联了原分支名）
git checkout 原feat分支
git reset --hard 新分支名  # 把原分支指向干净的新分支
git push --force-with-lease  # 因为只有你操作，安全
```

这样你就不用重新提 MR 了，原 MR 的历史会自动变干净。**当然，如果懒得折腾，直接用新分支提 MR 也完全 OK。**

---

### 🛡️ 防坑指南：rebase 的正确打开方式

你这次其实是**合并方向对了，但策略选错了场景**：

| 场景 | 推荐策略 | 原因 |
|------|---------|------|
| **把 dev 更新到本地 feat** | `git merge dev` (普通合并) | 保留历史痕迹，方便回滚，log 清晰 |
| **feat 开发完成，准备合回 dev** | `git rebase dev` (先) + merge (后) | 让 feat 的提交排在最上面，形成线性历史 |

**下次更新代码时：**
```bash
# 如果你只是想"拉取最新 dev 代码到我的分支"
git fetch origin
git merge origin/dev  # 不要用 rebase，除非你很确定

# 或者更保险：先切到 dev 更新，再切回 feat 合并
git checkout dev && git pull
git checkout feat/xxx
git merge dev  # 这样只会多一个 merge commit，好认也好回滚
```

**检查小技巧：** 执行任何合并/变基前，看一眼 `git log --oneline | wc -l`（提交数量）。如果突然从 5 个变成 50 个，**赶紧 Ctrl+C 还来得及！** 🚨
