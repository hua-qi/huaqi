### 1. `bridge` 对象是什么？
`bridge` 是一个名为 `CardBridge` 的接口实例，它是**卡片与宿主环境（以及 Agent）进行持续通信的桥梁**。因为卡片是独立的 UI 组件（运行在沙箱或者作为独立模块加载），它不能直接调用外部的网络请求，也不能直接控制 Agent。因此宿主环境会将这个 `bridge` 对象作为 Props 注入给卡片组件。

`CardBridge` 主要提供了以下 4 个核心方法：
*   **`queryData` (查询数据)**：卡片用来执行 SQL 或调用 Function 获取数据。比如图表组件请求真实数据、或者下拉框请求候选值。它直接和后端通信，结果返回给卡片自己渲染，不惊动 Agent。
*   **`executeAction` (执行操作)**：卡片调用后端的业务 Function 执行写操作（如点击提交按钮）。同样结果返回给卡片自己。
*   **`notifyAgent` (通知 Agent)**：卡片主动向大模型发信号（类似于对标 MCP Apps 的 `sendPrompt`）。比如表单提交成功后，卡片调用这个方法告诉 Agent：“表单提交成功了”，从而触发 Agent 进行下一步回复或操作。
*   **`reportResult` (上报渲染结果)**：卡片告诉宿主“我渲染完了，核心数据是这些”。宿主会将这些数据组装成 `Tool Return` 的结构抛给 Agent，让大模型知道卡片里呈现了什么内容。

### 2. CLI 是模块联邦（Module Federation）架构吗？
**是的**。
根据文档中的配置结构和架构说明：
*   在卡片资源（`CodeBundleConfig`）的数据模型中，明确存储了 **`remoteEntry`** 和 **`remoteName`** 字段，并且文档注释明确标出它就是 **Module Federation 入口**。
*   现有的 `ai-cli` 在执行 `release` 时，打包输出和上传到 CDN 的核心产物也是 `remoteEntry.js`。
*   这意味着研发本地通过 CLI 开发的是一个独立的模块，打包成 Webpack Module Federation 的 Remote 产物上传云端。而在用户界面的宿主环境里（ChatBot 界面），通过提取 CDN 链接动态加载（Dynamic Import）这个组件执行渲染。

[2026-04-03 16:37:00] [assistant]: 在了解了代码卡片（前端手写代码）之后，理解 **DSL 引擎** 就很简单了，它是专门为**数据卡片**服务的前端核心模块。

### 1. 什么是 DSL？
**DSL（Domain Specific Language，领域特定语言）**。在我们的系统中，它具体指的是那份名为 `construction_dsl` 的 JSON 配置。
比如，Agent 自动生成的数据卡片背后没有一行 JS 代码，只是一段 JSON 描述：
```json

  "card_type": "chart",
  "data_source": { "type": "sql", "query": "SELECT * FROM xxx" },
  "charts": [{ "chart_type": "bar", "x_axis": "dept" }]
}
```
这段 JSON 告诉系统：“我需要一个柱状图，数据来自这段 SQL，X轴是部门”。

### 2. DSL 引擎是什么？
**DSL 引擎** 就是内置在**前端宿主环境（ChatBot Web端）**里的一个**解析与渲染程序（或者叫低代码/无代码渲染器）**。
因为浏览器只能看懂 HTML/JS/CSS，看不懂上面那段 JSON。DSL 引擎的作用，就是把上面这串毫无生气的 JSON 文本，**“翻译”并渲染成一个真正可以点击、有交互的图表界面**。

### 3. DSL 引擎的具体工作流程
当宿主环境收到一条渲染“数据卡片”的指令时，DSL 引擎会接管工作：

1. **解析配置（Parse）：** 读取 `construction_dsl`，识别出这是一张“图表（chart）”，包含了一个“部门”筛选项。
2. **注入参数（Inject）：** 把 Agent 传过来的入参（比如 `params: { dept: "技术部" }`）填入筛选项作为默认值。
3. **请求数据（Fetch）：** 引擎代替卡片去向后端发起真实的数据请求（执行 DSL 里配置的 SQL 或 Function）。
4. **组装组件（Render）：** 拿到数据后，引擎从平台的标准组件库里，拿出标准的“筛选框组件”、“柱状图组件”，拼装在一起，渲染在页面上。
5. **自动上报（Report）：** 卡片画完后，DSL 引擎会默默提取核心数据，**自动**调用 `bridge.reportResult({ summary, data })` 返回给 Agent（这就是为什么前面提到，数据卡片的上报模式 `mode="auto"` 开发者完全无需关心）。

### 总结对比

* **对于“代码卡片”：** 是前端工程师用 React 写的代码，打包成了 JS 文件。浏览器拉下来直接执行渲染。不需要引擎“翻译”。
* **对于“数据卡片”：** 是 Agent 用 JSON 拼出来的积木图纸。浏览器看不懂，必须交由 **DSL 引擎** 根据图纸动态把积木（标准组件）拼装起来渲染。

3. **黑盒化与调试难度**：自动总结和截断机制虽然好用，但在某些场景下可能会默默丢弃大模型真正需要的关键细节信息；且深度绑定 LangChain 生态，抽象层较厚，排查底层问题通常强依赖 LangSmith 等可视化工具。

在 Agent 的核心逻辑（如 ReAct 模式：思考 -> 行动 -> 观察 -> 思考）中，大模型本身是无法直接执行代码或发起网络请求的。所谓的“调用”，本质上是大模型输出了一段**“暂停指令”**。

具体机制和两者的区别如下：

### 1. 机制本质：为什么必须暂停？
当大模型决定调用某个能力（无论 Tool 还是 Skill）时，它会输出一段特定的结构化数据（如 Function Calling 的 JSON）。
OneAgent 引擎在解析到这个意图时，必须**立即暂停（挂起）大模型的推理生成**。因为大模型需要拿到这个工具的**执行结果（Observation/Tool Return）**，才能基于真实数据进行下一步的思考或回答用户。如果不暂停，大模型就会陷入幻觉（自己编造工具的返回结果）。

### 2. 调用 Skill 时的暂停（后端闭环）
*   **执行链路**：Agent 发出指令 -> **引擎暂停 Agent** -> 引擎在后端（或沙箱中）执行对应的业务逻辑代码（比如查询库存 API）-> 拿到 JSON 数据 -> 将结果追加到对话上下文中 -> **唤醒并恢复 Agent**。
*   **特点**：这个暂停是非常短暂且纯服务端的。对于用户来说，几乎感觉不到中断，整个过程是**毫秒或秒级**的程序级等待。

### 3. 调用 Tool 时的暂停（端云异步闭环）
以 `show_a_card` 或 `create_card` 为例：
*   **执行链路**：Agent 发出指令 -> **引擎暂停 Agent** -> 引擎将指令通过 WebSocket 或接口下发给**前端客户端（用户的聊天界面）** -> 前端加载组件、请求数据、渲染界面 -> 前端执行完毕，调用 `CardBridge.reportResult` 上报数据 -> 引擎收到前端传回的 Tool Return -> **唤醒并恢复 Agent**。
*   **特点**：这个暂停跨越了“云端”和“客户端”。它需要等待前端完成 UI 渲染甚至等待用户的某些交互。如果网络慢或卡片加载久，Agent 会一直处于被挂起的等待状态（通常系统会设置一个超时时间的兜底机制，比如 10秒没收到前端的回报，就自动向大模型抛出一个超时错误）。

**总结：**
Agent 调用任何外部能力，在架构上都是一个**“挂起当前上下文 -> 移交控制权执行 -> 接收结果 -> 恢复上下文继续推理”**的异步过程。Tool 和 Skill 的区别不在于是否暂停，而在于**去哪里执行（前端 UI 宿主 vs 后端服务器）**以及**等待的时间长短**。
