[2026-03-30 14:52:59] [assistant]: Spec Driven Development (SDD) 的核心开发流程通常包含以下几个步骤：

1. **编写规范 (Write Spec)**：在写代码前，先定义清晰的 API 规范、数据结构或系统行为（如使用 OpenAPI/Swagger、GraphQL Schema 或设计文档）。
2. **评审与对齐 (Review & Align)**：前后端团队或利益相关者对规范进行评审，达成一致，将其作为契约 (Contract)。
3. **生成与模拟 (Generate & Mock)**：基于规范自动生成类型定义 (Types)、接口代码 (Stubs) 和 Mock 数据，前后端可以并行开发。
4. **编码实现 (Implement)**：开发者根据生成的接口和契约编写具体的业务逻辑代码。
5. **验证测试 (Verify & Test)**：使用规范自动生成测试用例或进行契约测试 (Contract Testing)，确保实现与规范完全一致。

这种流程将规范作为单一事实来源 (Single Source of Truth)，能有效减少沟通成本和集成问题。

[2026-03-30 15:02:39] [assistant]: 在 Spec Driven Development (SDD) 中，各个阶段的核心任务如下：

**0. 准备阶段 (Brainstorm & Requirements)**
*   收集业务需求和用户故事。
*   梳理业务流程，确定初步的数据模型和交互方式。

**1. 编写规范 (Write Spec)**
*   使用 OpenAPI、GraphQL、Protobuf 等标准定义 API 规范。
*   详细定义请求参数、响应数据结构、状态码及错误信息。
*   提供基础的示例文档。

**2. 评审与对齐 (Review & Align)**
*   前端、后端、QA 共同对齐规范细节。
*   评估技术可行性和业务覆盖度。
*   确认无误后锁定规范，将其作为前后端的“唯一契约 (Single Source of Truth)”。

**3. 生成与模拟 (Generate & Mock)**
*   **工具生成**：基于规范自动生成前端的数据模型 (如 TypeScript 类型) 和请求代码，以及后端的路由和接口骨架。
*   **启动 Mock**：基于规范自动生成 Mock Server，前端在后端真实接口开发好之前，直接对接 Mock 数据进行 UI 和交互开发。

**4. 编码实现 (Implement)**
*   **前端**：根据 Mock 数据完成页面渲染、状态管理和交互逻辑。
*   **后端**：在生成的接口骨架中填充业务逻辑（如数据库读写、第三方服务调用等）。
*   *此阶段前后端完全并行，互不阻塞。*

**5. 验证测试 (Verify & Test)**
*   **契约测试 (Contract Testing)**：验证后端的真实接口是否100%符合之前定义的规范（如使用 Dredd 或 Pact）。
*   集成测试：前后端真实联调，确保业务链路贯通。
*   如果发现规范需要调整，必须回到“第1步（编写规范）”进行修改，严禁绕过规范直接改代码。
