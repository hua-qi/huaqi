## ReAct 循环核心机制

### 整体流程图
```
StartEvent
  ↓ start()
  └─ 初始化系统Prompt、加载历史(最近3轮)
  ↓ MainAgentLoopEvent
  ↓ main_agent_loop() ◄─────────────────┐
  │                                      │
  ├─ depth >= 10? → GiveUpEvent          │
  ├─ LLM推理 → 解析<reason>展示思考过程  │
  ├─ 解析<tool>标签提取工具调用          │
  │   ├─ tool == 'summary'               │
  │   │   └─ 流式生成最终答案 → FinalAnswerEvent
  │   └─ tool == 'get_resources'         │
  │       或 'search_content_...'        │
  │       └─ 执行工具 → 结果追加消息 ───┘
  ↓ FinalAnswerEvent
  ↓ final_answer() → TaskAgentResultEvent
  ↓ end() → StopEvent
```

### LLM 响应格式约定（XML 协议）
LLM 必须输出结构化 XML，ReAct 循环靠正则解析驱动：
```xml
<reason>
  分析用户问题，需要先获取会话中的文件资源...
</reason>

<tool>
  <name>search_content_in_the_resources</name>
  <arguments>{"query": "销售额", "resources": [{"type": "file", "resource_id": "xxx"}]}</arguments>
</tool>
```
或者结束时：
```xml
<tool>
  <name>summary</name>
  <arguments>{}</arguments>
</tool>
```

### 两个工具
| 工具 | 作用 |
|------|------|
| `get_resources` | 获取当前会话中所有上传的文件列表 |
| `search_content_in_the_resources` | 在指定文件中做语义搜索（搜索后走 `CompressContextWorkflow` 压缩上下文） |
| `summary` | 伪工具，触发 LLM 流式生成最终答案 |

### 历史消息管理
- 跨轮对话历史存 MySQL（`agent_message_service`），每次启动加载
- 本轮循环历史存 `ctx`，每次 `main_agent_loop` 追加新消息
- `_extract_recent_history()` 从历史中截取最近 **3 轮**完整对话，避免超过上下文窗口

### 关键设计细节
- **资源类型强制修正**：不管 LLM 返回什么 `type`，一律修正为 `'file'`（防幻觉）
- **无工具调用时的容错**：检测到 LLM 响应没有 `<tool>` 标签，往消息里追加提示 `"请使用正确的格式"` 后继续循环
- **json_repair**：解析 `<arguments>` 时用 `json_repair.loads` 容错 LLM 输出的不合法 JSON
- **keepalive**：每 5 秒发一个 `PingEvent` 防止 SSE 连接超时

代码跑完后检查两件事：

2. **有没有打印数据表？**（有没有输出 `<dataframe>` 数据）

只要有一项不满足，就把错误原因告诉 AI：

> "你的代码跑完没生成图表，请检查是不是忘了调用 display_xxx_to_user()"

AI 看到错误后重新写，**最多给3次机会**。

---

**第四步：4 张图同时并行处理**

不是一张一张等，而是最多 4 张图同时让 AI 写代码、同时跑，互不等待，速度更快。

---

**第五步：汇总结果，AI 写一段总结**

所有图都处理完后，把成功的图、失败的图整理好，再让 AI 写一段 Markdown 文字总结，告诉用户"这个看板分析了什么、发现了什么"，最后一起返回给前端展示。

## 可交互看板的实现方式

### 核心答案：**交互性来自 ECharts 原生 HTML，通过 iframe 加载**

整个链路就4步：

---

**第一步：Python 代码在沙箱里生成 HTML 文件**

AI 写的代码调用 `display_standard_plot_to_user(html, "图表标题")`，这个函数内部用 pyecharts 把图表渲染成一个**完整的 HTML 文件**（包含 ECharts JS），存到文件存储（Blob/S3），拿到一个 `artifact_file_id`（UUID）。

---

**第二步：后端把 HTML 文件地址封装成 HtmlCard**

```
SandboxArtifacts
  artifact_file_id = "abc-123"
  artifact_type = "HTML"
  display_title = "各地区销售额对比"
        ↓
HtmlCard
  content.loadUrl = "https://cdn.xxx.com/abc-123.html"
  content.displayTitle = "各地区销售额对比"
  if_artwork = true   ← 标记为可交互作品
```

---

**第三步：前端用 iframe 加载这个 HTML**

```
前端收到 HtmlCard → 渲染一个 iframe
iframe src = loadUrl（指向那个 HTML 文件）
HTML 文件里内嵌 ECharts JS → 自动初始化图表
```

---

**第四步：交互能力完全来自 ECharts 本身**

- 鼠标悬停 → Tooltip 自动显示
- 点击图例 → 隐藏/显示某系列
- 框选/滚轮 → DataZoom 缩放

---

### 所以"可交互"的范围

| 能做 | 不能做 |
|------|--------|
| ECharts 图内悬停/缩放/图例筛选 | 跨图表联动（点A图筛选B图） |
| 每张图独立交互 | 实时修改查询参数重新拉数据 |
| 单图重绘（发新消息触发AI重新生成） | 拖拽改变图表布局 |

**本质上**：交互看板 = 若干个独立的 ECharts HTML 文件，每个放在一个 iframe 里，靠 ECharts 自身的交互能力，跟后端无关。

```
    → 验证两件事：
        ✓ 有 HTML artifact 产出
        ✓ stdout 里有 <dataframe> 数据输出
      失败 → 把错误告诉 LLM 重写，最多3次

步骤3：汇总
  成功的图 + 失败的图 → LLM 生成 Markdown 总结文字
```

### 第四阶段：组装响应，前端渲染

```
后端组装：
  MarkdownCard（总结文字）
  + HtmlCard × N（每张图的 loadUrl 指向对应 HTML 文件）
  + CodeAndConsoleCard（代码和执行日志）
  → 通过 SSE 流式推送给前端

前端渲染：
  Markdown 文字直接展示
  每个 HtmlCard → 渲染一个 iframe
  iframe 加载 HTML 文件（内嵌 ECharts JS）
  → 图表自动初始化，支持悬停/图例/缩放等原生交互
```
