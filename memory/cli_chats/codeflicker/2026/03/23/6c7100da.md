[2026-03-23 15:05:38] [assistant]: ## 难点一：经验质量

**核心矛盾：** 自动沉淀 vs 人工审核的取舍

### 方案：分层质量体系

```
自动候选 → 轻量人工确认 → 质量评分动态调整
```

**自动沉淀的触发信号（可信度从高到低）：**
- Agent 修复同一类错误超过 N 次 → 高可信，自动沉淀
- 用户接受了 agent 的代码且未在 X 天内回滚 → 中可信，待确认
- 用户显式标记"这个方案很好" → 最高可信，直接入库

**关键设计：** 不要追求"完美入库"，而是引入**置信度分**。低置信度的经验注入时降低权重或仅供参考，而不是完全过滤掉。

**冷启动破局：** 第一批经验可以由人工从历史 PR/commit 中提取，批量导入作为种子数据。

---

## 难点二：检索精度

**核心矛盾：** 语义相似 ≠ 场景适用

### 方案：多维度召回 + 适用性过滤

**召回不只用语义相似度，加入结构化过滤：**

```
检索条件 = 语义相似度(0.6权重) 
         + 技术栈标签匹配(必选过滤) 
         + 模块/文件路径相似(0.2权重)
         + 经验历史命中率(0.2权重)
```

**防止错误注入的关键机制：**
- 每条经验记录**被注入后用户是否采纳**，采纳率低于阈值的经验自动降权或下架
- 注入时对 agent 说明"这是历史经验，需判断是否适用"，而非"你必须这样做"——让 agent 自己做适用性判断，而不是盲目执行

---

## 难点三：时效性

**核心矛盾：** 代码库在变，经验库在老化

### 方案：经验绑定代码版本 + 主动失效检测

**每条经验存储时关联：**
```json
{
  "related_files": ["src/utils/request.ts"],
  "file_hash_at_creation": "abc123",
  "last_validated": "2026-01-10"
}
```

**失效检测策略：**
- 监听代码库变更（git hook 或定时扫描），若 `related_files` 发生结构性变化，自动将经验标记为"待复核"
- 经验被注入但 agent 最终没有采用时，累计记录，超过阈值触发人工复核
- 设置 TTL（如 90 天未被命中的经验自动归档），避免僵尸经验占用召回名额

**轻量复核机制：** 不要求人工重新写经验，只需回答"还适用吗？"，降低维护成本。

---

## 难点四：粒度控制

**核心矛盾：** 太细碎片化，太粗无法落地

### 方案：三层粒度分层存储

```
L1 - 模式层（架构/设计模式）   → 粗，适合跨模块复用
L2 - 场景层（具体功能实现方式）→ 中，主力层
L3 - 修复层（具体 bug + fix） → 细，高精度但低泛化
```

**不同层在注入时的用途不同：**
- L1 在任务开始时注入（告知项目约定）
- L2 在 agent 开始写具体功能时注入
- L3 在 agent 遭遇报错/异常时才注入（避免噪音）

**粒度判断自动化：** 入库时用 LLM 对经验做分类，打上层级标签，不需要人工判断粒度。

---

## 整体架构小结

```
代码生成事件
    ↓
[检索层] 多维召回 + 适用性过滤
    ↓
[注入层] 按 L1→L2→L3 分时机注入
    ↓
Agent 生成代码
    ↓
[反馈层] 采纳率统计 → 动态调整经验权重
    ↓
[监控层] 文件变更检测 → 失效标记
```

四个难点本质上是**一个质量飞轮问题**：初期靠人工种子数据 + 严格过滤保证质量，中期靠采纳率反馈自动优化，长期靠失效检测防止腐化。飞轮转起来之前是最难的阶段。

## 为什么不应该由 codeAgent 提供方做

codeAgent 提供方（如 Cursor、Claude 等）天然缺少两个东西：

1. **业务上下文** - 他们不知道你的项目约定、技术栈偏好、历史决策
2. **反馈闭环** - 他们看不到代码最终有没有被用、有没有被回滚

提供方能做的只是通用能力（如更好的基础模型、更准的语义检索），但"什么是这个项目的最佳实践"这个判断，**永远只有业务工程师自己知道**。

---

## 为什么不应该完全由业务工程师从零做

维护成本太高。如果每个团队都要自己实现向量存储、检索注入、失效检测这套基础设施，重复造轮子，且大多数团队没有这个精力。

---

## 合理的分工边界

| 层 | 由谁做 | 内容 |
|---|---|---|
| 基础设施层 | codeAgent 提供方 | 向量存储、检索注入接口、反馈采集 SDK |
| 经验内容层 | 业务工程师 | 种子数据、质量审核、失效判断 |
| 触发规则层 | 业务工程师配置 | 哪些场景注入、注入哪个粒度 |

**类比：** 就像 CI/CD 平台由基础设施团队提供，但流水线规则由业务团队自己写。工具归平台，知识归业务。
