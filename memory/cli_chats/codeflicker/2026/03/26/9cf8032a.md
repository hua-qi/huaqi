### 1. 后端：SSE 流式推送

后端使用 **SSE (Server-Sent Events)** 协议，将 AI 回答按 token 块逐步推送给前端，事件格式分三类：
- `event: update` — 流式中间数据（包含新增文字块）
- `event: end` — 流式结束
- `event: error` — 错误信息

---

### 2. 前端接收：fetchEventSource

`src/utils/fetchStream.ts` 用 `@microsoft/fetch-event-source` 封装 SSE 请求，注册三个回调：

```
onDataUpdate → 收到新数据块 → 追加到 Store
onDataEnd   → 流结束 → 保存完整消息
onError     → 异常处理
```

---

### 3. 状态管理：Zustand Store 合并策略

`src/stores/useStreamStore.ts` 负责将每一块增量数据合并：

| 卡片类型 | 合并方式 |
|---------|---------|
| `MarkdownCard` | 字符串 `+=` 逐字拼接 |
| `Thinking` | 按工具类型智能合并（爬虫/知识库/消息） |
| 新卡片（index 变化）| 追加新卡片对象 |

---

### 4. 渲染链路（打字机效果核心）

```
SSE 数据 → appendStreamResponse（增量追加）
         → scheduleFlush（RAF 批处理，每帧只刷一次）
         → streamResponse（Zustand 订阅）
         → MarkdownChunksCard（按段落+空行安全拆分，保护代码块）
         → CachedMarkdownCard（逐段落渲染）
```

打字机感来源：每帧 flush 一次 Store → React 重新渲染 → Markdown 内容变长 → 产生逐字增长的视觉效果。

---

### 5. 关键性能优化

- **RAF 批处理**：高频 SSE 事件不直接触发 React 更新，而是在下一帧统一 flush
- **段落级拆分**：避免一个超长 Markdown 组件整体重渲染
- **生成中禁用引用解析**：减少计算开销

### 1. 卡片效果实现

打字机的"卡片"效果由 **段落分割 + 逐段渲染** 实现，分三层：

**`MarkdownChunksCard`** 用 `splitMarkdownSafe()` 把完整 Markdown 内容按两种边界切分：
- 空行（段落边界）
- ` ``` ` 代码块（整体不拆，保证代码块完整）

每次流式数据更新（`cardContent.content` 增长），重新切分后 map 成多个 `CachedMarkdownCard`，新增段落自然追加在末尾，视觉上就是逐段"弹出"。

```
content: "段落A\n\n段落B\n\n```\ncode\n```"
           ↓ splitMarkdownSafe
chunks:  ["段落A", "段落B", "```\ncode\n```"]
           ↓ 渲染三个 CachedMarkdownCard
```

`ThinkingCard` 里对新出现的段落加了 CSS 动画：

```less
@keyframes fadeIn {
  from { opacity: 0; transform: translateY(4px); }
  to   { opacity: 1; transform: translateY(0); }
}
.is-new { animation: fadeIn 0.4s ease-in; }
```

### 2. Markdown 链接中间态处理

**结论：项目没有专门处理不完整链接的逻辑。**

`useStreamStore` 中 MarkdownCard 合并策略是纯字符串拼接：
```typescript
lastCardContent.content = lastCardContent.content.concat(newCardContent.content);
```

依赖两个机制规避中间态问题：
1. **`disableCitations={isGenerating}`** — 生成中禁止引用渲染，降低中间态暴露风险
2. **段落级分割的天然保护** — `splitMarkdownSafe` 以空行分段，链接通常在同一段落内，完整的段落才会被独立渲染

但如果链接跨越了段落边界（极少见），仍会出现 `[text](htt` 这样的破损文本。标准修复方式是在分割前加正则检测：

```typescript
// 检测末尾是否有未闭合的链接
const hasIncompleteLink = (text: string) => /\[[^\]]*\]\([^)]*$/.test(text);
// 若末尾有不完整链接，推迟分割，等待下一段补完
```

### 3. 防止整个组件重渲染

**三层防御：**

| 层级 | 组件 | 方式 | 状态 |
|------|------|------|------|
| 顶层 | `StreamUI` | `React.memo` + 3 个 `useMemo` | 已实现 |
| 中间层 | `MarkdownChunksCard` | 无 | **缺失** |
| 底层 | `CachedMarkdownCard` | `React.memo` | 已实现 |

**核心防重渲染逻辑：**

- `CachedMarkdownCard` 用 `React.memo` 包裹，只要 `content` / `index` 等 props 不变，就不重渲染。流式更新时旧的段落 props 不会变，只有最后一个段落的 content 在增长，所以 **只有最后一段重渲染**。
- key 用 `${cardContent.index}+${arrayIndex}` 组合键，稳定不变，保证已渲染段落的 DOM 不被销毁重建。
- RAF 批处理（`scheduleFlush`）确保每帧最多触发一次 Store flush，而非每个 SSE 事件都触发 React 更新。

**`MarkdownChunksCard` 缺少 `React.memo` 是个遗漏**，每次 StreamUI 重渲染都会重新执行它，建议补上：
```typescript
export default React.memo(MarkdownChunksCard);
```

2. **Render（渲染）**：把 AST 节点映射成对应的 HTML 标签
   ```
   heading → <h1>
   strong  → <strong>
   link    → <a>
   code    → <pre><code>
   ```

**豆包用的是这类库：**
- `marked` / `markdown-it`（转成 HTML 字符串，再 `dangerouslySetInnerHTML`）
- `react-markdown`（直接把 AST 节点映射成 React 组件，输出 div/p/span 等）

浏览器不认识 Markdown 语法，只认识 HTML。MD 只是一种便于书写的标记约定，必须经过解析才能在浏览器里展示成有样式的内容。

你们项目里的 `MarkdownCard` 组件本质上也是封装了这个解析+渲染过程。

[2026-03-26 15:18:53] [user]: MCP 协议长什么样子？
[2026-03-26 15:19:14] [assistant]: MCP（Model Context Protocol）是 Anthropic 提出的标准协议，用于让 AI 模型与外部工具/数据源通信。

### 传输层

基于 **JSON-RPC 2.0**，两种传输方式：
- **stdio**：本地进程间通信（AI 启动子进程，通过 stdin/stdout 交换消息）
- **SSE**：远程 HTTP 服务（和你们项目的流式回包同一个机制）

---

### 消息格式

**请求（Host → Server）**
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "tools/call",
  "params": {
    "name": "read_file",
    "arguments": {
      "path": "/tmp/foo.txt"
    }
  }
}
```

**响应（Server → Host）**
```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": [
      {
        "type": "text",
        "text": "文件内容..."
      }
    ]
  }
}
```

---

### 核心能力三类

| 能力 | method | 说明 |
|------|--------|------|
| **Tools** | `tools/list` `tools/call` | 让 AI 调用函数 |
| **Resources** | `resources/list` `resources/read` | 暴露文件/数据给 AI 读取 |
| **Prompts** | `prompts/list` `prompts/get` | 预定义提示词模板 |

---

### 完整握手流程

```
Host（AI）          Server（MCP插件）
   │                      │
   │── initialize ────────►│  协商版本、能力
   │◄─ result ─────────────│
   │                      │
   │── tools/list ─────────►│  获取可用工具列表
   │◄─ [{name, description, inputSchema}] ─│
   │                      │
   │  (AI 决定调用工具)     │
   │                      │
   │── tools/call ─────────►│  执行工具
   │◄─ {content} ──────────│  返回结果
```

---

### 本质

MCP 就是给 AI 定义了一套**标准插件接口**，让任何工具（数据库、文件系统、API）都能用同一种协议接入，AI 不需要知道工具内部实现，只需要按协议调用。你现在用的 codeflicker 里的 `read`、`grep`、`bash` 这些工具，底层就是走 MCP 协议暴露给我的。

### LLM 怎么"决定"调用哪个工具

完全靠 **tool 的 description** 和 **inputSchema** 来理解工具用途：

```json
{
  "name": "read_file",
  "description": "读取文件内容，适合查看代码、配置文件",
  "inputSchema": {
    "type": "object",
    "properties": {
      "path": { "type": "string", "description": "文件绝对路径" }
    }
  }
}
```

LLM 看到 description 后自行判断：当前任务需不需要这个工具、传什么参数。

---

### 所以 description 写得好不好直接影响 agent 行为

| description 质量 | 结果 |
|----------------|------|
| 描述清晰、边界明确 | 工具被正确调用 |
| 描述模糊 | LLM 可能选错工具或不调用 |
| 没有 description | 基本靠猜 |

这也是为什么 MCP 工具开发的核心工作之一就是写好 description，而不只是实现功能本身。
