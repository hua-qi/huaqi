[2026-03-29 01:34:56] [assistant]: ### 当前文档结构的优劣势

**优势：**
- 分层清晰，有金字塔结构
- 有 ADR 记录架构决策（spec/decisions/）
- 覆盖面广（产品、技术、运维）

**劣势：**
- **ARCHITECTURE.md 重复**：根目录和 docs/ 各有一份，容易不同步
- **缺少 CHANGELOG**：没有记录版本变更和 bug 修复历史
- **docs/ 过于平铺**：13 个文件没有子目录分类，难以扩展
- **spec/architecture、spec/roadmap 是空目录**：预留但未使用，和根目录的 ROADMAP/ARCHITECTURE 职责重叠
- **没有明确的"维护规范"**：不知道什么改动需要更新哪个文档

### 修 Bug / 加 Feature 要记录吗？

**要记录，但记录的位置不同：**

| 改动类型 | 记录位置 | 内容 |
|---------|---------|------|
| Bug 修复 | `docs/CHANGELOG.md`（新建） | 版本号、日期、修复了什么 |
| 新 Feature（小） | `docs/CHANGELOG.md` | 简短说明新增了什么 |
| 新 Feature（大/影响架构） | `docs/features/` 下新建或更新对应文档 | 设计思路、接口、用法 |
| 涉及架构决策 | `spec/decisions/ADR-xxx.md` | 为什么这么做、有哪些备选方案 |

**最实用的建议是先建一个 `CHANGELOG.md`**，每次改动无论大小都在里面追加一行，这样成本最低且收益最高。
