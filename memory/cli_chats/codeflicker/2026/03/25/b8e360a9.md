[2026-03-25 16:11:36] [user]: 解释一下 BM25 关键词匹配
[2026-03-25 16:12:26] [assistant]: BM25（Best Match 25）是一种基于**词频**的文本检索算法，是 Elasticsearch 等搜索引擎的核心评分机制。

## 核心思想

```
文档相关性 = Σ(每个查询词的权重 × 词频得分)
```

相比简单词频计数，BM25 做了三个关键优化：

| 优化点 | 说明 | 示例 |
|--------|------|------|
| **TF 饱和** | 词出现次数越多，加分递减 | "React" 出现 10 次 vs 100 次，得分差距不大 |
| **IDF 降权** | 常见词（如 "the"）权重低，罕见词权重高 | "useEffect" 比 "function" 更能区分文档 |
| **长度归一化** | 长文档的词频被稀释 | 100 字文档出现 5 次 vs 1000 字文档出现 5 次，前者更相关 |

## 简化的评分公式

```
score(D, Q) = Σ IDF(q) * [ f(q,D) * (k1+1) ] / [ f(q,D) + k1 * (1-b + b*|D|/avgdl) ]
```

- `f(q,D)`：词 q 在文档 D 中出现次数
- `|D|`：文档长度
- `avgdl`：平均文档长度
- `k1=1.2, b=0.75`：经验参数

## 代码示例

```python

]

# 2. 查询（分词后）

| 特性 | BM25 | 向量检索 |
|------|------|----------|
| **可解释性** | ✅ 明确看到匹配了哪些词 | ❌ 黑盒，不知道为啥匹配 |

| **同义词** | ❌ "汽车"≠"车辆" | ✅ 语义相近都能召回 |
| **无模型依赖** | ✅ 纯算法 | ❌ 依赖 Embedding 模型 |

```

## 未来更可能的演进

随着大模型变强，**混合方案**才是主流，而非纯标签：

```
检索阶段          重排阶段
────────          ────────
标签过滤 ──┐
           ├──→ Top-20 召回 ──→ LLM 精排 ──→ Top-3 输出
BM25关键词 ─┘      （粗筛）      （理解语义相似性）
            ↑
        向量召回 ──┘
```

**极端情况**：如果未来大模型上下文无限长且推理成本极低，可以直接把所有经验扔给 LLM 让它自己选，那确实不需要任何检索。但在可预见的未来，检索仍然是必要的。

## 结论

> 纯标签 + 关键词方案的效果会随着大模型变强而**线性改善**，但不会**指数级提升**，因为标签的离散性限制了表达能力上限。向量检索的语义扩展能力在可见的未来仍然不可替代。

## 检索流程（模型无关）

```
用户查询（任何模型生成）
   ↓
1. 标签精确匹配过滤
   - tech_stack 交集
   - problem_type 匹配
   - scene 交集
   ↓
2. BM25 关键词排序
   - 对 "content" 字段做 BM25 评分
   - 关键词命中越多，排名越高
   ↓
3. LLM 重排（可选，可替换任何模型）
   - 把 Top-10 经验传给 LLM
   - 提示："判断哪条最符合当前问题，给出理由"
   ↓
返回结果（含匹配理由）
```

## 实现示例

```python
class ModelAgnosticSearch:
    def search(self, query: str, context_tags: list[str]) -> list[Experience]:
        # 1. 标签过滤（确定性规则）
        candidates = [
            exp for exp in self.experiences
            if any(t in exp.metadata.tech_stack for t in context_tags)
        ]
        
        # 2. BM25 排序（纯算法，无模型依赖）
        tokenized_corpus = [self._tokenize(exp.content.problem) for exp in candidates]
        bm25 = BM25Okapi(tokenized_corpus)
        scores = bm25.get_scores(self._tokenize(query))
        
        # 按分数排序
        ranked = sorted(zip(candidates, scores), key=lambda x: x[1], reverse=True)
        
        # 3. LLM 精排（任何模型都可替换）
        top_candidates = [exp for exp, _ in ranked[:10]]
        final_results = self.llm.rerank(query, top_candidates)  # 可换任何 LLM
        
        return final_results
```

## 兼容性的本质

**向量方案的兼容性问题是数据层的**：
- 历史数据用模型 A 嵌入
- 新查询用模型 B 嵌入  
- 两者不匹配 → 检索失效

**结构化文本方案的兼容性在应用层**：
- 数据是纯文本，任何模型都能读
- 检索逻辑是确定性的，不依赖模型
- LLM 只用于可选的重排，可随意替换

class UnifiedSearch:

        # 优先：标签+BM25（模型无关）
        results = self.tag_search(query, tags)
        
        # 后备：向量检索（如果启用且标签无结果）
