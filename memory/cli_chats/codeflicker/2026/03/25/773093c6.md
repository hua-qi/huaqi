[2026-03-25 16:03:43] [assistant]: ## TCP 协议详解

### 核心特点
- **面向连接**：通信前需建立连接（三次握手）
- **可靠传输**：确认应答、重传机制、流量控制、拥塞控制
- **字节流**：不保留报文边界，保证数据顺序

### 三次握手（建立连接）
```
客户端 → SYN(seq=x) → 服务端
客户端 ← SYN(seq=y, ack=x+1) ← 服务端  
客户端 → ACK(ack=y+1) → 服务端
```
**为什么是三次？** 防止历史重复连接初始化，同步双方初始序列号

### 四次挥手（断开连接）
```
客户端 → FIN → 服务端
客户端 ← ACK ← 服务端
客户端 ← FIN ← 服务端（数据发完后）
客户端 → ACK → 服务端
```
**TIME_WAIT 状态**：等待 2MSL，确保最后一个 ACK 被收到，防止旧连接数据包干扰新连接

### 可靠传输机制
| 机制 | 说明 |
|------|------|
| 确认应答 | ACK 确认收到，带序号 |
| 超时重传 | 定时器超时未收到 ACK 则重传 |
| 快速重传 | 收到 3 个重复 ACK 立即重传 |
| 滑动窗口 | 批量发送，提升效率 |
| 流量控制 | 接收窗口（rwnd）控制发送速率 |
| 拥塞控制 | 慢启动、拥塞避免、快重传、快恢复 |

## HTTP 协议详解

### 版本演进
| 版本 | 特点 |
|------|------|
| HTTP/1.0 | 短连接，每次请求新建 TCP 连接 |
| HTTP/1.1 | **持久连接**（keep-alive）、管道化、分块传输 |
| HTTP/2 | **二进制分帧**、多路复用、头部压缩（HPACK）、服务器推送 |
| HTTP/3 | 基于 **QUIC**（UDP），解决队头阻塞，内置 TLS 1.3 |

### 常见方法
- **GET**：获取资源，幂等，可缓存
- **POST**：提交数据，非幂等
- **PUT**：替换资源，幂等
- **DELETE**：删除资源，幂等
- **PATCH**：部分更新

### 状态码
| 分类 | 常见码 |
|------|--------|
| 2xx 成功 | 200 OK, 201 Created, 204 No Content |
| 3xx 重定向 | 301 永久, 302 临时, 304 缓存 |
| 4xx 客户端错误 | 400 Bad Request, 401 未认证, 403 禁止, **404** |
| 5xx 服务端错误 | 500 内部错误, 502 网关错误, **503** 服务不可用 |

### 缓存机制
```
Cache-Control: max-age=3600  // 强缓存
ETag + If-None-Match         // 协商缓存
Last-Modified + If-Modified-Since
```

### HTTPS = HTTP + SSL/TLS
1. **TLS 握手**：协商加密算法，交换密钥（非对称加密）
2. **数据传输**：对称加密
3. **证书验证**：CA 公钥验证服务器证书链

## TCP vs HTTP 关系

```
应用层:    HTTP / HTTPS
传输层:    TCP  (可靠传输)
网络层:    IP   (寻址路由)
链路层:    Ethernet / WiFi
```

**HTTP 基于 TCP**：HTTP 请求前需先完成 TCP 三次握手，HTTP 响应后可通过 keep-alive 复用 TCP 连接

## 为什么存在服务端

| 特性 | 说明 |
|------|------|
| **存储位置** | 服务端 GPU 高带宽显存（HBM）|
| **数据形式** | 浮点数矩阵（K、V 向量），非原始文本 |
| **生命周期** | 单次对话会话，超时或结束后释放 |
| **内存占用** | 与对话长度成正比（长对话可能占数 GB）|

## 工作机制

```
第一轮："你好"
  ↓ 计算得到 token[你]、token[好] 的 K、V 向量 → 存入 KV-Cache
  
第二轮："请介绍自己"  
  ↓ 只需计算新 token 的 Q，直接与缓存中的 K 做 Attention
  ↓ 避免重新编码 "你好"，节省 90%+ 计算量
```

## 容易混淆的概念

| 概念 | 存储位置 | 内容 | 用途 |
|------|----------|------|------|
| **KV-Cache** | 服务端 GPU | 模型中间层向量 | 加速推理，避免重复计算 |
| **对话历史** | 服务端数据库 | 原始文本 messages | 持久化记录，多轮上下文 |
| **本地缓存** | 客户端 | 已完成的历史回复 | 离线查看，减少请求 |

**关键区别**：对话历史是"输入文本"，KV-Cache 是"计算好的中间结果"。服务端收到历史文本后会重新填充 KV-Cache（或从缓存恢复），用于增量生成。

[2026-03-25 16:42:53] [assistant]: ## Prefill 阶段详解

### 一句话定义
**并行计算输入提示（Prompt）所有 token 的 Key/Value 向量**，为后续自回归生成建立缓存。

### 具体计算过程

```
输入: "请翻译: Hello" (4个token: [请, 翻译, :, Hello])

Prefill 阶段:
┌─────────────────────────────────────────┐
│  Token:    请      翻译      :     Hello │
│     ↓      ↓        ↓       ↓       ↓   │
│  Embedding + Positional Encoding         │
│     ↓      ↓        ↓       ↓       ↓   │
│  ┌─────────────────────────────────────┐ │
│  │  Transformer Block × N (并行计算)    │ │
│  │  - Self-Attention: Q×K^T → Softmax  │ │
│  │  - 每层输出: hidden states           │ │
│  │  - 提取每层的 K、V → KV-Cache       │ │
│  └─────────────────────────────────────┘ │
│     ↓      ↓        ↓       ↓       ↓   │
│  最终输出: 4个位置的 KV 向量存入显存      │
└─────────────────────────────────────────┘

Decode 阶段:
  输入: "" (空，或开始标记)
  利用 KV-Cache 快速计算，逐个生成 token
```

### 核心特点

| 特性 | Prefill | Decode |
|------|---------|--------|
| **计算模式** | 并行（矩阵乘法） | 串行（无法并行）|
| **算力需求** | 高（计算密集型） | 低（内存带宽密集型）|
| **延迟来源** | 矩阵乘法计算时间 | 显存读取 KV-Cache 时间|
| **优化目标** | 算力利用率（MFU） | 带宽利用率 |

### 为什么需要 Prefill

1. **建立 KV-Cache**：后续每个新 token 都要与所有历史 token 做 Attention
2. **避免重复计算**：用户输入只处理一次，生成阶段复用
3. **确定初始状态**：提供第一个输出 token 的 hidden state

### 性能瓶颈与优化

| 问题 | 优化方案 |
|------|----------|
| 长 Prompt 计算慢 | **Chunked Prefill**：分块流水，边预填充边解码 |
| 首 token 延迟高 | 减少层数、量化（INT8/INT4）、投机采样 |
| 显存占用大 | PageAttention（vLLM）、KV-Cache 压缩 |

### 实际数据（以 LLaMA-70B 为例）

- 输入 4096 tokens 的 Prefill：~100-500ms（取决于硬件）
- 输出 1 个 token 的 Decode：~20-50ms
- **长输入场景**：Prefill 延迟可能超过后续全部生成时间

**面试常考点**：为什么首字回复慢？答：Prefill 需要完整计算输入，且无法流式输出。

    messages = [{"role": "user", "content": user_input}]
    
    while True:
        response = llm.chat(messages, tools=tools)
        msg = response.choices[0].message
        
        if not msg.tool_calls:  # 没有工具调用，直接回答
            return msg.content
            
        # 有工具调用，执行并追加结果
        for tool_call in msg.tool_calls:
            result = execute_tool(tool_call)
            messages.append({
                "role": "tool",
                "tool_call_id": tool_call.id,
                "content": str(result)
            })

## 关系梳理

| 层级 | 作用 | 示例 |
|------|------|------|
| **CoT** | 单步推理的**思考方式** | "让我先分析…然后…所以答案是…" |
| **Planning** | 多步任务的**全局规划** | "完成目标需要3步：①…②…③…" |

**CoT 是 Planning 的实现基础之一**，但 Planning 还涉及：
- **任务分解**（Task Decomposition）
- **反思修正**（ReAct、Reflexion）
- **长期规划**（ToT、GoT）

---

## 具体实现方式

### 1. 标准 CoT → 简单规划
```
Prompt: "让我们一步步思考..."
输出: 推理链自然形成执行顺序
局限: 无法回滚，一步错步步错
```

### 2. ReAct（CoT + Action）→ 动态规划
```
Thought: 我需要先查天气
Action: 调用天气API
Observation: 25度
Thought: 现在可以推荐穿衣了...
```
**特点**：每步根据观察调整计划，是 CoT 的交互式扩展

### 3. ToT（Tree of Thoughts）→ 多路径规划
```
      [目标]
     /   |   \
  [方案A][方案B][方案C]  ← 并行生成多个计划
    |      |      |
  评估   评估   评估    ← LLM 自我评估
    \      |      /
      选择最优路径
```
**特点**：显式维护规划树，可回溯

### 4. 结构化规划（LLM Compiler / Plan-and-Solve）
```
Step 1: [工具A] → 输出作为 Step 2 输入
Step 2: [工具B] → ...
```
**特点**：先一次性生成 DAG 计划图，再执行

---

## 总结

| 场景 | 方法 |
|------|------|
| 简单顺序任务 | Zero-shot CoT |
| 需观察反馈 | **ReAct**（最常用）|
| 复杂多路径 | ToT / GoT |
| 确定性流程 | PDDL / 工作流引擎 |

**面试一句话**：Planning 通常基于 CoT 实现推理过程，但高级 Agent 会引入**树搜索、反思、外部规划器**来增强可靠性。

## Code Agent vs 通用 Agent

| 维度 | 通用 Agent | Code Agent |
|------|-----------|------------|
| **环境** | 网页、API、数据库 | **代码仓库、文件系统、终端** |
| **感知** | 文本回复、JSON 数据 | **代码结构、错误堆栈、测试结果** |
| **工具** | 搜索、计算器、天气 API | **文件读写、代码执行、Git 操作** |
| **行动** | 生成文本、调用服务 | **修改代码、运行命令、提交 PR** |
| **反馈** | 人类评价 | **编译器/测试通过或失败** |

## Code Agent 核心特征

```
Observation          Action
   ↑                   ↓
错误日志 ← 终端执行  编辑文件
   ↑                   ↓
测试结果 ← 代码运行  读写代码
   ↑                   ↓
代码结构 ← 静态分析  运行测试
```

**关键区别**：Code Agent 有**可验证的反馈回路**——代码能否编译、测试是否通过是客观标准，而通用 Agent 的"正确性"往往需人工判断。
