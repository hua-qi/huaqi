[2026-04-09 14:48:52] [assistant]: ## Module Federation 运行时：主应用加载远端组件的流程

### 1. 初始化阶段：加载 remoteEntry.js

主应用通过配置的 `remote` 地址，动态插入 `<script>` 标签加载远端的入口文件：

```js
// webpack 内部等价逻辑
const script = document.createElement('script');
script.src = 'http://remote-app/remoteEntry.js';
document.head.appendChild(script);
```

`remoteEntry.js` 执行后，会在 `window` 上挂载一个**容器对象**（Container），key 为 remote 名称：

```js
window['remoteApp'] = {
  init(shareScope) { ... },   // 初始化共享作用域
  get(moduleName) { ... },    // 获取暴露的模块
}
```

---

### 2. 共享作用域初始化：`init()`

主应用调用 `container.init(shareScope)`，将**已加载的共享模块**（如 React、ReactDOM）注册到共享作用域，避免远端重复加载：

```
主应用 shareScope = { react: { '18.0.0': { get, loaded: true } } }
         ↓ init()
远端容器检查自己需要的 react 版本是否满足
→ 满足则复用主应用的，不满足则加载自己的
```

---

### 3. 获取模块：`container.get(moduleName)`

```js
const factory = await window['remoteApp'].get('./Button');
const Module = factory();
```

`get()` 内部：
1. 检查该模块的 chunk 是否已加载
2. 未加载则动态 `import()` 对应的异步 chunk
3. 返回模块工厂函数（factory）

---

### 4. React 使用侧的懒加载

```js
const RemoteButton = React.lazy(() =>
  import('remoteApp/Button')
);
```

webpack 将 `import('remoteApp/Button')` 编译为上述 `init + get` 的完整调用链，整个过程对开发者透明。

---

### 完整调用链

```
import('remoteApp/Button')
  │
  ├─ 1. 加载 remoteEntry.js（若未加载）
  ├─ 2. container.init(sharedScope)  ← 协商共享依赖
  ├─ 3. container.get('./Button')    ← 触发异步 chunk 加载
  └─ 4. factory()                    ← 返回模块导出
```

---

### 关键点

| 机制 | 作用 |
|---|---|
| `remoteEntry.js` | 远端的"索引"，暴露 init/get 接口 |
| `shareScope` | 共享依赖版本协商，防止多实例 |
| `container.get()` | 按需加载实际业务 chunk |
| 动态 `<script>` 注入 | 实现运行时（非构建时）远端加载 |

# Module Federation 完整使用教程

## 一、核心概念

Module Federation 将应用分为两种角色：

| 角色 | 说明 | 配置关键字 |
|---|---|---|
| **Provider（生产者）** | 对外暴露模块 | `exposes` |
| **Consumer（消费者）** | 消费远端模块 | `remotes` |

一个应用可以同时是生产者和消费者。

---

## 二、安装

**Webpack 项目：**
```bash
npm i @module-federation/enhanced
```

**Rsbuild 项目：**
```bash
npm i @module-federation/rsbuild-plugin
```

---

## 三、Provider（生产者）配置

`webpack.config.js`：

```js
const { ModuleFederationPlugin } = require('@module-federation/enhanced/webpack');

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'provider',          // 全局唯一名称
      exposes: {
        '.': './src/index.tsx',               // 默认导出
        './Button': './src/components/Button.tsx',
        './utils': './src/utils/add.ts',
      },
      shared: {
        react: { singleton: true },
        'react-dom': { singleton: true },
      },
    }),
  ],
};
```

构建后会生成 `remoteEntry.js` 或 `mf-manifest.json`（推荐用 manifest）。

---

## 四、Consumer（消费者）配置

`webpack.config.js`：

```js
const { ModuleFederationPlugin } = require('@module-federation/enhanced/webpack');

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'consumer',
      remotes: {
        // 别名: 'provider的name@entry地址'
        'myProvider': 'provider@http://localhost:3000/mf-manifest.json',
      },
      shared: {
        react: { singleton: true },
        'react-dom': { singleton: true },
      },
    }),
  ],
};
```

---

## 五、消费远端模块

配置好 `remotes` 后，直接 `import` 即可：

```tsx
// 静态导入
import Button from 'myProvider/Button';

// 懒加载（推荐）
const Button = React.lazy(() => import('myProvider/Button'));

function App() {
  return (
    <React.Suspense fallback="Loading...">
      <Button />
    </React.Suspense>
  );
}
```

---

## 六、shared 共享依赖详解

shared 的作用：**避免 React 等依赖被加载两次**（尤其是有单例要求的库）。

```js
shared: {
  // 数组格式（简单场景）
  // shared: ['react', 'react-dom'],

  // 对象格式（精细控制）
  react: {
    singleton: true,        // 只允许一个版本存在
    requiredVersion: '^18.0.0',  // 版本要求
    eager: false,           // true=打包进入口(体积大，慎用)
  },
  'react-dom': {
    singleton: true,
    requiredVersion: '^18.0.0',
  },
}
```

| 字段 | 说明 |
|---|---|
| `singleton` | 强制单例，版本不一致时加载更高版本 |
| `requiredVersion` | 声明所需版本范围 |
| `eager` | `true` 时同步加载，否则异步按需加载 |
| `shareScope` | 隔离共享池，默认 `'default'` |

---

## 七、remotes 的两种 entry 格式

```js
remotes: {
  // 推荐：manifest.json（支持类型提示、预加载、devtool）
  'provider': 'provider@http://localhost:3000/mf-manifest.json',

  // 传统：remoteEntry.js
  'provider': 'provider@http://localhost:3000/remoteEntry.js',
}
```

---

## 八、快速创建新项目

```bash
# 交互式创建
npm create module-federation@latest

# 直接指定参数
npx create-module-federation -d my-app -t provider-rsbuild -n my_provider
```

---

## 九、完整最小示例

**项目结构：**
```
provider-app/   → 暴露 Button 组件，运行在 :3000
consumer-app/   → 消费 Button 组件，运行在 :4000
```

**provider** `webpack.config.js`：
```js
new ModuleFederationPlugin({
  name: 'provider',
  exposes: { './Button': './src/Button.tsx' },
  shared: { react: { singleton: true } },
})
```

**consumer** `webpack.config.js`：
```js
new ModuleFederationPlugin({
  name: 'consumer',
  remotes: { provider: 'provider@http://localhost:3000/remoteEntry.js' },
  shared: { react: { singleton: true } },
})
```

**consumer** 使用：
```tsx
const Button = React.lazy(() => import('provider/Button'));
```

---

## 十、关键注意事项

1. **`name` 必须全局唯一**，且 `remotes` 中 `@` 前的名称必须与对方的 `name` 一致
2. **`exposes` 的 key 不能用 `"./"`**，默认导出用 `"."`
3. **有单例要求的依赖**（React、Router、Store）必须加 `singleton: true`，否则可能出现多实例 Bug
4. **推荐用 `mf-manifest.json`** 而非 `remoteEntry.js`，支持动态类型提示和 devtool 调试
[2026-04-09 15:17:30] [user]: 对于生产者，每个组件都有对应的 remoteEntry 链接吗
[2026-04-09 15:17:52] [assistant]: 不是的。

**一个 Provider 应用只有一个 `remoteEntry.js`（或 `mf-manifest.json`）**，它是整个应用的统一入口索引。

```
http://localhost:3000/remoteEntry.js   ← 只有这一个入口
```

`remoteEntry.js` 本身不包含组件代码，它只是一个**目录/索引**，记录了所有 `exposes` 的模块映射：

```
remoteEntry.js
  ├── './Button'  → chunk_button.js   (真正的组件代码)
  ├── './Modal'   → chunk_modal.js
  └── './utils'   → chunk_utils.js
```

Consumer 消费时的流程：

```
1. 加载 remoteEntry.js（一次）
2. import('provider/Button')  → 按需加载 chunk_button.js
3. import('provider/Modal')   → 按需加载 chunk_modal.js
```

所以 Consumer 配置里只需要写一个地址：

```js
remotes: {
  provider: 'provider@http://localhost:3000/remoteEntry.js',
  //                    ↑ 只需要这一个，所有暴露的组件都从这里索引
}
```

**总结**：`remoteEntry` 是应用级别的，不是组件级别的。每个组件对应的是独立的 **chunk 文件**，由运行时按需动态加载。

[2026-04-09 16:28:17] [assistant]: ## 什么是 SDK

SDK（Software Development Kit，软件开发工具包）是第三方服务提供给开发者的**一套现成代码集合**，封装了与该服务交互的所有复杂逻辑，让你不需要自己处理底层细节。

比如：
- 你想接入微信支付 → 微信提供支付 SDK
- 你想用地图功能 → 高德/Google 提供地图 SDK
- 你想做数据分析 → 神策/Mixpanel 提供埋点 SDK

**本质**：SDK = 别人帮你写好的封装层，你只调接口。

---

## 前端如何安装和使用 SDK

### 方式一：npm 包（最常见）

```bash
npm install some-sdk
```

```js
import SDK from 'some-sdk';

const client = new SDK({ apiKey: 'your-key' });
client.doSomething();
```

### 方式二：CDN 脚本（无需构建工具）

```html
<script src="https://cdn.example.com/sdk.min.js"></script>
<script>
  // 挂载在 window 上
  window.SomeSDK.init({ apiKey: 'your-key' });
</script>
```

### 方式三：动态加载脚本

```js
const script = document.createElement('script');
script.src = 'https://cdn.example.com/sdk.min.js';
script.onload = () => window.SomeSDK.init({ apiKey: 'your-key' });
document.head.appendChild(script);
```

---

## 核心区别

| | npm 包 | CDN 脚本 |
|---|---|---|
| 适用场景 | 有构建工具（webpack/vite） | 纯 HTML 或需要全局注入 |
| Tree Shaking | 支持 | 不支持 |
| 版本管理 | `package.json` 锁定 | URL 手动维护 |

现代前端项目优先用 **npm 包**，埋点/监控类 SDK 常用 **CDN 脚本**（因为需要尽早执行，不依赖构建流程）。

# A2A 协议完整字段解析

## 一、顶层核心对象总览

```
AgentCard         → Agent 的"身份证"，供客户端发现和初始化
  └── AgentSkill  → Agent 能力列表
  └── AgentCapabilities → 支持的协议特性

Message           → 单轮通信单元
  └── Part[]      → 内容容器（文本/文件/结构数据）

Task              → 有状态的工作单元
  ├── TaskStatus  → 当前状态
  └── Artifact[]  → 产出物
       └── Part[] → 内容容器
```

```json
  "provider": {

    {
      "uri": "https://example.com/ext/v1",

    }
  ]
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `name` | string | 是 | Agent 名称 |

## 三、Part（内容容器）

Message 和 Artifact 的最小内容单元，每个 Part **只能包含一种内容字段**：


### Part 字段说明

| 字段 | 类型 | 说明 |
|---|---|---|
| `text` | string | 纯文本内容（四选一） |
| `raw` | string | Base64 编码的二进制数据（四选一） |
| `url` | string | 外部文件 URI（四选一） |
| `data` | object | 结构化 JSON 数据（四选一） |
| `mediaType` | string | MIME 类型，如 `"image/png"` |
| `filename` | string | 文件名 |
| `metadata` | object | 附加 key-value 元信息 |

## 四、Message（消息）

单轮通信单元，角色为 `user` 或 `agent`。

```json
{
  "role": "user",
  "messageId": "msg-001",
  "contextId": "ctx-abc",
  "taskId": "task-xyz",
  "referenceTaskIds": ["task-prev-001"],
  "parts": [
    { "text": "Plan a trip to Tokyo for 5 days." }
  ],
  "metadata": {}
}
```

### Message 字段说明

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `role` | `"user"` \| `"agent"` | 是 | 消息发送方角色 |
| `messageId` | string | 是 | 消息唯一 ID |
| `contextId` | string | 否 | 所属会话上下文 ID |
| `taskId` | string | 否 | 关联的 Task ID |

## 五、Task（任务）

有状态的工作单元，用于跟踪长时运行操作。

```json

  "contextId": "ctx-abc",

  },
```

### Task 字段说明

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | Task 唯一 ID |
| `contextId` | string | 所属会话上下文 ID |
| `status` | TaskStatus | 当前状态对象 |
| `artifacts` | Artifact[] | 产出物列表 |
| `history` | Message[] | 历史消息（可选） |
| `metadata` | object | 附加元信息 |

### TaskStatus 字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `state` | TaskState | 当前状态值（见下表） |
| `message` | Message | 附带的 Agent 消息 |
| `timestamp` | string | 状态变更时间（ISO 8601） |

### TaskState 枚举值

| 状态 | 类型 | 说明 |
|---|---|---|
| `submitted` | 进行中 | 已提交，等待处理 |
| `working` | 进行中 | 处理中 |
| `input-required` | **中断** | 需要用户补充输入 |
| `auth-required` | **中断** | 需要用户授权 |
| `completed` | **终止** | 成功完成 |
| `canceled` | **终止** | 已取消 |
| `rejected` | **终止** | 被拒绝 |
| `failed` | **终止** | 执行失败 |

## MCP-UI 内容通过 A2A 协议传递

核心思路：MCP-UI 的 `UIResource`（一段 HTML）本质是**二进制/文本内容**，完全可以用 A2A 的 `Part` 来承载。

---

### 映射关系

| MCP-UI 字段 | A2A 对应位置 | 说明 |
|---|---|---|
| `resource.text`（HTML 字符串） | `Part.text` 或 `Part.raw` | 内联传递 HTML |
| `resource.blob`（Base64 HTML） | `Part.raw` | Base64 直接对应 |
| `resource.uri`（`ui://...`） | `Part.url` + `metadata.uiUri` | URI 引用方式 |
| `resource.mimeType` | `Part.mediaType` | `"text/html;profile=mcp-app"` |
| `_meta.ui.resourceUri` | `Part.metadata.resourceUri` | 工具与 UI 的绑定关系 |

---

### 方案一：内联 HTML（rawHtml 模式）

将 HTML 内容直接放进 A2A Message 的 Part：

```json
{
  "jsonrpc": "2.0",
  "method": "message/send",
  "params": {
    "message": {
      "role": "agent",
      "messageId": "msg-001",
      "parts": [
        {
          "text": "这是工具的执行结果",
          "mediaType": "text/plain"
        },
        {
          "raw": "<base64 encoded HTML>",
          "mediaType": "text/html;profile=mcp-app",
          "filename": "widget.html",
          "metadata": {
            "uiUri": "ui://my-server/widget",
            "mcpUiType": "rawHtml"
          }
        }
      ]
    }
  }
}
```

---

### 方案二：外部 URL 引用（externalUrl 模式）

HTML 托管在远端，A2A 只传 URL：

```json
{
  "parts": [
    {
      "url": "https://my-server.com/widget.html",
      "mediaType": "text/html;profile=mcp-app",
      "metadata": {
        "uiUri": "ui://my-server/widget",
        "mcpUiType": "externalUrl"
      }
    }
  ]
}
```

---

### 方案三：结构化 data Part（最完整）

把整个 UIResource 结构作为 `data` 传递，保留所有 MCP-UI 语义：

```json
{
  "parts": [
    {
      "data": {
        "type": "resource",
        "resource": {
          "uri": "ui://my-server/widget",
          "mimeType": "text/html;profile=mcp-app",
          "text": "<html>...</html>"
        }
      },
      "mediaType": "application/json",
      "metadata": {
        "protocol": "mcp-ui",
        "version": "1.0"
      }
    }
  ]
}
```

---

### 完整交互流程

```
Consumer（A2A Client）               Provider（A2A Server / MCP Server）
        │                                         │
        │── message/send ──────────────────────▶ │
        │   { parts: [{ text: "渲染 Widget" }] }  │
        │                                         │
        │ ◀─── Task(completed) ───────────────── │
        │   {                                     │
        │     artifacts: [{                       │
        │       parts: [{                         │
        │         raw: "<base64 HTML>",           │
        │         mediaType: "text/html;          │
        │                  profile=mcp-app",      │
        │         metadata: {                     │
        │           uiUri: "ui://server/widget"   │
        │         }                               │
        │       }]                                │
        │     }]                                  │
        │   }                                     │
        │                                         │
  客户端解析 Part，检测 mediaType               │
  → 识别为 MCP-UI 内容                          │
  → 渲染进沙箱 iframe                           │
```

---

### 关键判断点

客户端收到 A2A Artifact 后，通过以下条件识别是否为 MCP-UI 内容：

```js
const isMcpUI = part.mediaType === 'text/html;profile=mcp-app'
             || part.metadata?.protocol === 'mcp-ui'
             || part.metadata?.uiUri?.startsWith('ui://')
```

**总结**：MCP-UI 没有专属传输协议，它的内容（HTML + 元数据）完全可以通过 A2A `Part` 的 `raw`/`url`/`data` + `mediaType` + `metadata` 组合来承载，**由双方约定 `metadata` 中的字段作为识别标识**。

### 方式一：动态 import() + 运行时注入（官方推荐）

不在构建时写死 remotes，而是运行时手动加载：

```js
async function loadRemote(url, scope, module) {
  // 1. 动态注入 script
  await new Promise((resolve, reject) => {
    const script = document.createElement('script');
    script.src = url;
    script.onload = resolve;
    script.onerror = reject;
    document.head.appendChild(script);
  });

  // 2. 初始化容器
  await __webpack_init_sharing__('default');
  const container = window[scope];
  await container.init(__webpack_share_scopes__.default);

  // 3. 获取模块
  const factory = await container.get(module);
  return factory();
}

// 使用
const Button = await loadRemote(
  'http://localhost:3001/remoteEntry.js',
  'providerA',
  './Button'
);
```

---

### 方式二：`@module-federation/enhanced` 运行时 API

```js
import { init, loadRemote } from '@module-federation/enhanced/runtime';

// 运行时动态注册（可从接口获取配置）
const remoteConfig = await fetch('/api/remotes').then(r => r.json());

  name: 'consumer',
  remotes: remoteConfig.map(item => ({
    name: item.name,
    entry: item.url,  // 动态 URL
  })),
});

```
