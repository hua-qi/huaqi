```typescript
{
  type: 'resource',
  resource: {
    uri: 'ui://<component-name>/<instance-id>',  // ui:// 开头
    mimeType: 'text/html;profile=mcp-app',       // 固定值
    text?: string,   // encoding='text' 时，HTML 字符串
    blob?: string,   // encoding='blob' 时，Base64 编码的 HTML
    _meta?: object,  // 元数据
  }
}
```

关键点：
- `type` 固定为 `"resource"`
- `mimeType` 固定为 `"text/html;profile=mcp-app"`，客户端用此识别 MCP-UI 资源
- `text` 和 `blob` 二选一，取决于服务端 `encoding` 配置
- 工具本身通过 `_meta.ui.resourceUri` 指向该 URI，client 再通过 `resources/read` 拉取完整的 UIResource

[2026-04-14 16:44:15] [user]: AppRenderer 拿到  _meta.ui.resourceUri 后再发一次 resources/read 这个 Case 是考虑的什么场景呢？
[2026-04-14 16:44:39] [assistant]: 这个模式（`_meta.ui.resourceUri` + `resources/read`）主要考虑以下场景：

**1. UI 模板与数据分离**
Tool 调用返回的是业务数据，UI 模板是相对静态的。同一个 `ui://my-card/template` 可以被多次工具调用复用，不需要每次都把整段 HTML 序列化进 tool response。

**2. 大体积 HTML**
工具调用走的是 LLM 上下文，HTML 如果很大（含内联 CSS/JS），直接内嵌会浪费 token。用 URI 引用，让 Client 走单独通道拉取。

**3. 动态/按需生成资源**
`registerAppResource` 的处理器可以在 `resources/read` 时才根据当前状态动态生成 HTML，而不是在 tool 调用时就固定下来。适合资源内容依赖时间、用户状态等运行时因素的场景。

**4. 资源可独立刷新**
Client 可以在不重新调用 tool 的情况下，直接再次 `resources/read` 来刷新 UI，适合需要轮询或局部更新的卡片。

---

简单总结：**inline HTML 适合轻量、一次性渲染；resourceUri 模式适合模板复用、大体积内容、或需要独立刷新的场景。**
[2026-04-14 16:50:23] [user]: 依据 MCP-UI 规范，我们是否应该在 MCP-UI Server 中调用业务接口，进而将数据拼装进 HTML 后，作为 ToolResult 一次性返回？
[2026-04-14 16:50:39] [assistant]: 根据 MCP-UI 规范的设计意图，**不应该**这样做。

规范推荐的职责划分是：

```
MCP-UI Server  →  只负责提供 UI 模板（HTML/CSS/JS）
Agent          →  调用业务接口，获取数据，作为 toolResult 注入卡片
MCP App (iframe) →  在客户端侧用 toolResult 数据渲染 UI
```

核心机制是 `initial-render-data` / `toolResult`：Host 通过 `ui/notifications/tool-result` 这条 postMessage 通道，把业务数据**推送进 iframe**，由卡片自己渲染。

**如果在 MCP-UI Server 里调用业务接口拼 HTML：**
- 违背了 UI 与数据分离的设计意图
- HTML 变成一次性快照，无法响应后续交互（如 iframe 内触发 `tools/call` 再次请求数据）
- MCP-UI Server 承担了本该属于 Agent 的业务编排职责，耦合过重
- 无法利用 `AppRenderer` 的 `toolResult` 注入机制，也无法做资源复用

**正确做法：**
1. Agent 调用业务接口拿到数据
2. Agent 把数据作为 `toolResult` 携带在响应里（如本仓库 `ext-mcp-ui-resource` 扩展中的 `toolResult` 字段）
3. MCP-UI Server 只返回静态/模板化的 HTML
4. `AppRenderer` 把 `toolResult` 通过 postMessage 注入卡片，卡片渲染数据

官方的设计实际上**没有严格规定业务数据必须由 Agent 处理、不能由 MCP Server 处理**。

官方数据流是：

```
MCP Server 的 tool 被调用
  → Server 处理业务逻辑，生成 UI 资源（含数据）
  → 返回 toolResult（包含 UIResource）
  → Client AppRenderer 拿到 toolResult，通过 ui/notifications/tool-result 注入 iframe
  → iframe 渲染
```

关键点：

- **MCP Server 就是负责业务逻辑的一方**（官方原文：*"Tools receive input parameters and return results containing UI resources"*）
- `toolResult` 是 AppRenderer 的 prop，用于把 tool 执行结果注入 iframe，**数据由 tool 本身产出**
- 官方并未要求 UI 模板与业务数据必须分离到不同服务

所以**在 MCP-UI Server 里调用业务接口、将数据拼进 HTML（rawHtml 模式）或通过 `toolResult` 携带（MCP Apps 模式）一起返回，是完全符合规范的做法**。

之前的结论"MCP-UI Server 不应调用业务接口"是基于架构分层推断，并非官方规范要求，需要纠正。
