[2026-04-14 19:43:03] [assistant]: ## Monorepo 配置

**工具**: pnpm workspace + NX

```yaml
# pnpm-workspace.yaml
packages:
  - 'websites/*'   # 7个前端应用
  - 'packages/*'   # 14个库包
  - 'servers/*'    # 后端服务
  - 'widgets/*'    # Widget库
  - 'config/*'     # 共享配置
```

NX 负责任务缓存和并行执行（build/test 等操作），缓存输出到 `dist/{projectRoot}`。

---

## Vite 配置

分三类：

**1. 库包 (`packages/*`)** — 多格式输出 `es/cjs/umd`，用 `vite-plugin-dts` 生成类型，外部化 react/react-dom：
```ts
build: { lib: { entry: 'components/index.ts', formats: ['es', 'cjs', 'umd'] } }
```

**2. 应用 (`websites/*`)** — 标准 SPA 配置，含 dev server 代理（多个后端）、LESS 主题变量、CDN 保护插件、环境变量注入、手动 chunk 分割：
```ts
server: { proxy: { '/java/api/workspace': target, '/magic-ai': { ws: true }, ... } }
```

**3. Widget (`widgets/*`)** — 只输出 UMD，多入口分开构建，外部化 react 及所有 @kaleido/* 包。

---

## TypeScript 配置

- 根 `tsconfig.json` — 通用配置 + 路径别名
- `tsconfig.vite.json` — 应用构建专用（`moduleResolution: Node`）
- `tsconfig.dumi.json` — 文档/库开发专用（`moduleResolution: NodeNext`）
- 各包继承最近的根 tsconfig，覆盖自己的 `paths`
[2026-04-14 19:46:04] [user]: NX 是什么？
[2026-04-14 19:46:13] [assistant]: NX 是一个 monorepo 构建系统和任务执行工具，主要解决大型多包项目的效率问题。

核心功能：
- **任务缓存**：记录 build/test 等任务的输出，未变更时直接命中缓存跳过执行
- **并行执行**：自动分析依赖图，最大化并行运行任务
- **增量构建**：只重新构建受代码变更影响的包
- **任务流水线**：定义任务间依赖关系（如 build 依赖上游包的 build）

在这个项目里，NX 主要作为任务调度层，实际构建工具仍是 Vite/tsc，NX 负责决定"哪些包需要重新跑、以什么顺序跑"。

[2026-04-14 19:53:57] [user]: 该项目是如何实现多应用协作的？
[2026-04-14 19:58:50] [assistant]: ## 多应用协作方式

### 1. PostMessage 双向通信（核心机制）

Workspace 与 Runtime 之间的主要通信方式：

```
Workspace 编辑 DSL
    ↓ 编译成 IR
    ↓ postMessage(ir, previewOrigin)
Runtime（iframe 或新窗口）
    ↓ 处理 DSL，更新页面渲染
    ↓ [HMR 场景] parent.postMessage({ type: 'getLatestData' })
Workspace 响应最新编译数据
```

- `websites/workspace/src/hooks/usePreviewInstances.ts` — Workspace 向所有预览实例广播
- `websites/runtime-v2/src/hooks/useHmrApp.ts` — Runtime 主动请求最新数据

### 2. Widget 动态加载（UMD）

Widget 库打包成 UMD 格式，运行时动态通过 `<script>` 注入到全局：

- **构建**：每个入口（`builder.ts`、`render.ts`、`meta.ts`）单独构建成 UMD
- **命名规则**：全局变量名为 `kaleido-widgets@builder@2_0_608`（防冲突）
- **加载**：`loadScript(url)` → 从 `window[name]` 取出注册到组件工厂
- Workspace 加载 `builder.js`，Runtime 加载 `render.js`，两侧各取所需

### 3. 全局对象共享依赖

两个应用的 `main.tsx` 都会把公共库挂到 `window`，供 UMD Widget 访问：

```typescript
Reflect.set(window, 'React', React);
Reflect.set(window, 'KaleidoWidgetCore', KaleidoWidgetCore);
// ...
```

### 4. SessionStorage 预览实例追踪

Workspace 通过 sessionStorage 持久化 Runtime 预览窗口的引用（`window.name` 机制），刷新后可恢复与已打开预览窗口的通信连接。

### 5. URL 路由跳转

Runtime 内部页面跳转通过 `appRouter.push()`，跨应用或外部链接通过 `getTopWindow().open()` / `location.href`（用 `window.parent` 递归找顶层窗口）。

---

**总结**：核心是 **PostMessage（编辑器↔运行时实时同步）+ UMD 动态加载（Widget 按需注入）**，无微前端框架，是自研的轻量协作方案。
