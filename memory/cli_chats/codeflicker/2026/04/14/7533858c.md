## 一、实时数据（SSE 流式处理）

### 整体数据流

```
后端 SSE → fetchEventSource → onmessage 回调
→ onDataUpdate(data) → Zustand + Immer 精确更新 → React re-render
```

### 核心实现细节（`fetchStream.ts`）

**1. 为什么用 `@microsoft/fetch-event-source` 而不是原生 EventSource？**

原生 `EventSource` 有两个硬伤：
- 只支持 GET，不能传 body（AI 接口需要 POST 传 prompt）
- 标签页不可见时会自动断连重连

`fetch-event-source` 两个都解决了，`openWhenHidden: true` 是关键——**切换到其他 tab 时连接不中断**，否则用户切回来会丢数据。

**2. 重试机制（注意代码里的细节）**

代码里写了最多 5 次重试的 while 循环，但实际上 `catch` 里直接 `throw error` 了，相当于**禁用了重试**。这是个有意的决定（注释掉的代码里有指数退避策略）。面试可以说：「考虑过重试，但 AI 流式接口出错时重试会导致重复生成，所以禁掉了，改为通知上层处理。」

**3. 三种 event 类型的处理**
```
update → 增量数据 → 调用 onDataUpdate
end    → 流结束   → 调用 onDataEnd
error  → 服务端错误 → throw，终止流
```

### 状态更新层（`useConversationStore.ts`）

这里面试亮点是 **Zustand + Immer 的精确更新**。

```typescript
updateSpecificCodeCardToCurrentConversation: (messageId, cardIndex, value) =>
  set(produce<State>((state) => {
    // 三层查找：消息 → ARTIFACT_CARD → CODE_CARD
    const message = state.currentConversation.messages.find(m => m.messageId === messageId);
    const artifactCard = message.cardContents?.find(c => c.cardType === 'ARTIFACT_CARD');
    const codeCard = artifactCard.content.cardContents.find(c => c.index === cardIndex);
    codeCard.content.code = value;  // 直接 mutation，Immer 处理不可变性
  }))
```

关键点：**流数据每次只更新某个具体 cardIndex 的 code 字段**，不是替换整条消息，所以渲染只触发那一个卡片组件的更新。

**面试问：为什么用 Immer 而不是手动展开？**

消息数据是三层嵌套结构（消息 → 卡片容器 → 子卡片），手写展开要写很多 `...spread`，且容易出错，Immer 的 `produce` 允许直接 mutation 且保证引用正确性（浅层未改变的引用不变，帮助 React.memo 做对比）。

## 二、微前端（Module Federation 动态卡片）

### 架构设计

```
主应用（app）
├── useDynamicRegisterCards    ← 初始化，注册所有远程模块
│   ├── GET /getCardConfig     ← 拉取 [{name, entry, widgetList}]
│   └── init({ remotes })      ← 配置 MF runtime
│
└── useDynamicImportCard       ← 按需加载单个组件
    └── loadRemote(`scope/widget`)  ← 动态 import
```

### 两个 Hook 的职责分工

**`useDynamicRegisterCards`**：应用启动时调一次，把所有卡片的远程地址注册进来。关键代码：
```typescript
init({
  name: 'app',
  remotes: data,  // 来自服务端的配置，热更新无需发版
  shared: {
    react: { singleton: true, ... },    // 防止多个 React 实例
    reactDom: { singleton: true, ... }
  }
});
```

**`useDynamicImportCard`**：用到某个 widget 时才加载，天然懒加载：
```typescript
const scope = cardLibraryList.find(p => p.widgetList?.includes(widget))?.name;
const result = await loadRemote(`${scope}/${widget}`);
```

先从 store 里找这个 widget 属于哪个 remote（scope），再拼成 `scope/widget` 去加载。

### 几个设计决策值得聊

**1. 为什么 React 要配置 singleton？**

多个远程模块都依赖 React，如果各自打包各自的 React，页面上会有多个 React 实例，Hooks 规则会被破坏（`useState` 找到错误的 fiber），所以必须指定 `singleton: true`，让所有模块共享主应用的 React。

**2. `localCards` URL 参数调试机制**

```typescript
const localCardsStr = searchParams.get('localCards');
if (localCardsStr?.startsWith('http')) {
  return res?.data?.result?.map((r) => {
    if (r.name === 'ihr_widget_provider') {
      return { ...r, entry: localCardsStr };  // 替换 entry 为本地地址
    }
    return r;
  });
}
```

开发卡片时在 URL 加 `?localCards=http://localhost:8080`，可以让主应用加载本地开发中的卡片，**不需要部署就能联调**。这是微前端开发体验的关键。

**3. 懒加载模式（`lazy` 参数）**

`useDynamicImportCard` 默认自动加载，但支持 `lazy=true`，此时需要手动调 `load()`。适合不在当前视口的卡片，用户滚动到时再加载，减少初始加载请求数。

轻量级 React 状态管理库，替代 Redux 的方案。

```typescript
const useStore = create((set) => ({
  count: 0,
  increment: () => set((state) => ({ count: state.count + 1 }))
}));

// 组件里用
const count = useStore((state) => state.count);
```

核心优势：比 Redux 少 90% 的样板代码，支持细粒度 selector（只订阅用到的字段，不触发多余 re-render）。

## Immer

解决 JavaScript **不可变数据**的库。

React/Redux 要求状态更新必须返回新对象，不能直接 mutation：
```typescript
// 原始方式（手写展开，嵌套越深越痛苦）
set(state => ({
  ...state,
  currentConversation: {
    ...state.currentConversation,
    messages: state.currentConversation.messages.map(msg =>
      msg.messageId === id ? { ...msg, content: newContent } : msg
    )
  }
}));

// Immer 的 produce（直接 mutation，Immer 内部生成新对象）
set(produce(state => {
  const msg = state.currentConversation.messages.find(m => m.messageId === id);
  msg.content = newContent;  // 直接改，底层用 Proxy 追踪变更
}));
```

Immer 用 **Proxy** 拦截所有 mutation 操作，最终生成一个新的不可变对象，未修改的节点引用保持不变（帮助 `React.memo` 做对比）。

```

         └── card.cardType: MajorCardType    ← 决定渲染什么

### 第一层：有没有 `cardContents`

`MixCardWidgetRenderer` 直接遍历 `message.cardContents`，后端返回几个卡片就渲染几个，没有这个字段就什么都不渲染。

`ChatBotMessageContentRenderer` 里有一套 switch 逻辑（`renderAssistantCard`）：

cardType === 'NoteCard'      → 笔记卡片
cardType === 'SkillCard'     → 技能卡片

```

## 总结

前端几乎没有自己的渲染判断逻辑，所有决策都来自后端：
- **渲染什么类型** → `cardType`
- **渲染几个** → `cardContents` 数组长度
- **展示方式** → `displayPosition`（嵌入对话 or 折叠片段）

这种设计的好处是灵活——新增卡片类型、改变展示位置都不需要改前端代码（微前端动态加载配合），缺点是前端对渲染结果的控制力弱，调试时需要了解后端数据结构。

## 底层基础：ContentEditable + HTML 字符串

输入框用的是 `react-contenteditable`，它的 `html` prop 就是一段 HTML 字符串，state 里存的是：

```html
普通文字 <span class="highlight person" data-id="123" contenteditable="false">@张三</span>&nbsp; 继续输入
```

`contenteditable="false"` 让 @标签不可编辑，`&nbsp;` 确保光标能落到标签后面。所有"特殊 token"都是 HTML span/a 标签，普通文字就是裸文本。

## 光标位置计算的难点

`getCursorPosition` 要计算"可见字符"的光标位置，但 innerHTML 里有大量 HTML 标签：

```typescript
const getCursorPosition = (node) => {
  const range = selection.getRangeAt(0);
  const preCaretRange = range.cloneRange();
  preCaretRange.selectNodeContents(node);
  preCaretRange.setEnd(range.endContainer, range.endOffset);

  const tmp = document.createElement('div');
  tmp.appendChild(preCaretRange.cloneContents()); // 克隆光标前的所有 DOM

  // innerHTML 长度 = 光标前所有字符（含 HTML 标签字符）
  return tmp.innerHTML.replace(/&nbsp;/g, ' ').length - selected;
};
```

## 原生 contenteditable

HTML 里任何元素加上 `contenteditable="true"` 就变成可编辑区域：

```html
<div contenteditable="true">用户可以直接在这里输入</div>
```

和 `<input>` / `<textarea>` 的区别是：它的内容是**真实 DOM**，可以包含 HTML 标签，而不只是纯文本。

---

## 为什么不用 input/textarea？

`<input>` 和 `<textarea>` 的值只能是纯字符串，无法在里面渲染有样式的 chip 标签。

项目里需要这种效果：
```
普通文字 [@张三] [/深度思考] 继续输入...
```

其中 `[@张三]` 是一个不可编辑的蓝色 span 标签，这只有 contenteditable 能做到。

---

## react-contenteditable 做了什么

原生 contenteditable 在 React 里有一个经典问题：React 的 Virtual DOM diff 会和用户直接操作 DOM 产生冲突，导致光标跳动、输入卡顿。

`react-contenteditable` 解决了这个问题，用法类似受控组件：

```tsx
<ContentEditable
  innerRef={inputRef}
  html={inputValue}          // 传入 HTML 字符串
  onChange={handleChange}    // e.target.value 是更新后的 HTML 字符串
  onKeyDown={handleKeyDown}
/>
```

本质上它在内部做了 `shouldComponentUpdate` 优化，只在 `html` prop 真正变化时才更新 DOM，避免 React 重新渲染时破坏光标位置。

## 一、文件加密上传（Web Crypto API）

`src/utils/fileUtils.ts`，用了混合加密方案：

```
RSA-OAEP 公钥加密 AES 密钥 → AES-GCM 分块加密文件内容
```

1. 用 `crypto.subtle.generateKey` 生成一次性 AES-256 对称密钥
2. 用服务端公钥（RSA-OAEP）加密这个 AES 密钥
3. 用 `file.stream().getReader()` 流式读取文件，分块（16KB）AES-GCM 加密
4. 把加密后的 AES 密钥 + 加密文件一起上传

**面试问点**：为什么不直接用 RSA 加密文件？——RSA 只适合加密小数据，文件太大性能极差，混合加密是标准做法。
