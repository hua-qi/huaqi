---

## MCP-UI 是什么

MCP-UI（现已标准化为 **MCP Apps**）是 MCP（Model Context Protocol）的官方交互式 UI 标准，允许 AI 工具构建富动态界面。

**核心特性：**
- **安全**：所有远程代码在沙箱 iframe 中执行
- **灵活**：支持 HTML 内容，兼容 MCP Apps 和传统 MCP-UI 宿主

---

## 规范主要规范了什么

MCP-UI 规范定义了 AI 工具如何通过 **资源（Resource）** 挂载 UI，核心机制是：
- Server 创建 `ui://` 协议的 UI 资源
- Tool 通过 `_meta.ui.resourceUri` 关联该资源
- Client 读取并渲染该资源到沙箱 iframe

---

## Server SDK 使用（`@mcp-ui/server`）

```typescript
import { registerAppTool, registerAppResource } from '@modelcontextprotocol/ext-apps/server';
import { createUIResource } from '@mcp-ui/server';

const widgetUI = createUIResource({
  uri: 'ui://my-server/widget',
  content: { type: 'rawHtml', htmlString: '<h1>Interactive Widget</h1>' },
  encoding: 'text',
});

// 注册 UI 资源
registerAppResource(server, 'widget_ui', widgetUI.resource.uri, {}, async () => ({
  contents: [widgetUI.resource]
}));

// 注册 Tool，关联 UI
registerAppTool(server, 'show_widget', {
  description: 'Show interactive widget',
  inputSchema: { query: z.string() },
  _meta: { ui: { resourceUri: widgetUI.resource.uri } }  // 关键：关联 UI
}, async ({ query }) => {
  return { content: [{ type: 'text', text: `Query: ${query}` }] };
});
```

---

## Client SDK 使用（`@mcp-ui/client`）

```tsx
import { AppRenderer } from '@mcp-ui/client';

<AppRenderer
  client={client}
  toolName={toolName}
  sandbox={{ url: sandboxUrl }}
  toolInput={toolInput}
  toolResult={toolResult}
  onOpenLink={async ({ url }) => window.open(url)}
  onMessage={async (params) => console.log('Message:', params)}
/>
```

`AppRenderer` 会自动从 tool 的 `_meta.ui.resourceUri` 读取资源并渲染到沙箱 iframe。

---

更多详情见 [mcp-ui.dev](https://mcp-ui.dev) 或 [GitHub](https://github.com/idosal/mcp-ui)。
[2026-04-14 17:26:13] [user]: MCP-UI 解决的是什么问题？
[2026-04-14 17:26:25] [assistant]: 基于已有的信息，MCP-UI 解决的核心问题是：

**MCP 工具（Tool）只能返回纯文本/结构化数据，无法呈现交互式 UI。**

具体来说：

- **传统 MCP Tool 的局限**：AI 调用工具后，结果只能以文本形式返回给模型再输出给用户，无法直接展示图表、表单、可交互组件等富界面
- **MCP-UI 的解法**：让 Tool 通过 `_meta.ui.resourceUri` 携带一个 UI 资源（HTML），宿主（Client）识别后将其渲染到沙箱 iframe，用户可以直接与 UI 交互

**典型场景：**
- AI 工具返回数据可视化图表（而不是一串数字）
- 表单填写交互（而不是多轮对话式收集参数）
- 富文本/媒体展示

本质上是在 MCP 协议层面补齐了 **"AI 工具调用结果的 UI 渲染能力"** 这一缺失环节。

[2026-04-14 17:54:58] [assistant]: 现在我对整个流程有了清晰的认识，来解答你的问题：

---

## Tool 绑定 `_meta.ui.resourceUri` 后的完整流程

### 1. `tool/call` 阶段 — 与普通 tool 完全相同

`_meta.ui.resourceUri` **不影响** `tools/call` 的请求和响应本身。Server 正常返回 `toolResult`（文本、数据等），`_meta` 只存在于 **tool 定义**（`tools/list` 的响应中），不在 `tool/call` 结果里。

### 2. Client 如何获取 UI

`AppRenderer` 在渲染时执行两步：

**Step 1：从 tool 定义中找 resourceUri**

```typescript
// 调用 tools/list，找到对应 tool，读取 tool._meta 中的 resourceUri
const info = await getToolUiResourceUri(client, toolName);
// 返回 { uri: 'ui://my-server/widget' }
```

**Step 2：通过 `resources/read` 拉取 HTML**

```typescript
const resource = await client.readResource({ uri: 'ui://my-server/widget' });
// 返回 mimeType 为 text/html 的内容
const html = resource.contents[0].text;
```

### 3. `toolResult` 如何传给 UI

HTML 渲染到沙箱 iframe 后，通过 **AppBridge**（基于 `postMessage`）将 `toolResult` 发送给 iframe 内的 Guest UI：

```
Host (AppRenderer)  --postMessage-->  沙箱 iframe (Guest UI)
                    toolResult 数据
```

Guest UI 通过监听 AppBridge 消息获取 `toolResult`，从而展示对应的可交互界面。

---

### 整体时序

```
用户触发 tool/call
    → Server 返回 toolResult（普通 MCP 响应）
    → AppRenderer 调用 tools/list 找到 tool._meta.ui.resourceUri
    → AppRenderer 调用 resources/read 拉取 HTML
    → HTML 注入沙箱 iframe
    → AppBridge postMessage 将 toolInput + toolResult 推送给 iframe
    → Guest UI 渲染交互界面
```

**关键点**：UI 资源加载（`resources/read`）和 tool 执行（`tools/call`）是**解耦的**，`_meta.ui.resourceUri` 只是告诉 Client "去哪里读 UI 的 HTML"，`toolResult` 通过 iframe 通信桥单独注入。

[2026-04-15 10:45:15] [assistant]: 从源码可以清晰看到答案：

**是的，使用 JSON-RPC 格式。**

AppBridge 继承自 MCP SDK 的 `Protocol` 类（`class AppBridge extends Protocol`），所有 iframe 间通信都遵循 JSON-RPC 2.0 格式，并使用 `postMessage` 传输。

具体消息类型包括：

| 方向 | 方法名 | 说明 |
|------|--------|------|
| Host → Guest | `ui/notifications/tool-result` | 推送 toolResult |
| Host → Guest | `ui/notifications/tool-input` | 推送 toolInput |
| Host → Guest | `ui/notifications/host-context-changed` | 主题/尺寸等上下文变更 |
| Host → Guest | `ui/notifications/sandbox-resource-ready` | HTML 内容就绪 |
| Guest → Host | `ui/open-link` | 请求打开链接 |
| Guest → Host | `ui/message` | 向模型上下文发消息 |
| Guest → Host | `ui/download-file` | 请求下载文件 |
| Guest → Host | `ui/request-display-mode` | 请求切换展示模式 |
| Guest → Host | `ui/notifications/size-changed` | 通知尺寸变化 |
| 双向 | `tools/call`, `resources/read` 等标准 MCP 方法 | Guest 可直接调用 MCP 方法，AppBridge 转发给 MCP Client |

本质上是在 iframe `postMessage` 通道上跑了一套完整的 MCP JSON-RPC 协议，Guest UI 可以直接调用 `tools/call`、`resources/read` 等标准 MCP 方法，AppBridge 负责转发给真正的 MCP Client。

[2026-04-15 10:59:30] [user]: tools/list 的作用是什么？
[2026-04-15 10:59:39] [assistant]: 在 MCP-UI 的 Guest UI 场景中，`tools/list` 的作用是：

**让 Guest UI 知道当前 MCP Server 暴露了哪些工具**，从而在 iframe 内部自行构建交互界面（比如动态生成表单、按钮、操作菜单等），然后再通过 `tools/call` 调用对应工具。

典型场景：
- Guest UI 是一个通用的"工具面板"，启动时调用 `tools/list` 动态渲染所有可用工具的操作入口
- 当 Host 推送 `notifications/tools/list_changed` 时，Guest UI 重新调用 `tools/list` 刷新界面

本质上，Guest UI 拥有与普通 MCP Client 相同的能力——它自己就是一个运行在 iframe 里的轻量 MCP Client，AppBridge 充当它与真实 MCP Server 之间的代理。
