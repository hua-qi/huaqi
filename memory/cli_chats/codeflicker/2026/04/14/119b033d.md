## The Process

### Phase 1: Understanding
- Check current project state in working directory
- Ask ONE question at a time to refine the idea

- **IMPORTANT: Use AskUserQuestion tool when asking clarification questions**

- Prefer multiple choice when possible
- Gather: Purpose, constraints, success criteria

### Phase 2: Exploration
- Propose 2-3 different approaches

- Ask your human partner which approach resonates

### Phase 3: Design Presentation
- Present in 200-300 word sections
- Cover: Architecture, components, data flow, error handling, testing
- Ask after each section: "Does this look right so far?"

## When to Revisit Earlier Phases

**You can and should go backward when:**
- Partner reveals new constraint during Phase 2 or 3 → Return to Phase 1 to understand it
- Validation shows fundamental gap in requirements → Return to Phase 1
- Partner questions approach during Phase 3 → Return to Phase 2 to explore alternatives
- Something doesn't make sense → Go back and clarify

**Don't force forward linearly** when going backward would give better results.

## Remember
- One question per message during Phase 1
- Apply YAGNI ruthlessly
- Explore 2-3 alternatives before settling
- Present incrementally, validate as you go
- Go backward when needed - flexibility > rigid progression
- Don't edit or write code during brainstorming

## Phase 2：探索方案

我先把这四个方向的**核心设计问题**和**与现有扩展的关系**拆开说：

---

### 当前扩展基线

现有的 `mcp_ui_resource` 扩展做了一件事：**在 A2A 响应里夹带一张卡片**。

```
A2A parts[] = [ text part, data part(kind=mcp_ui_resource) ]
```

接下来四个新扩展，都是在这个基础上**纵向深化**或**横向扩展**。

---

### 方向一：卡片内多轮对话

**当前状态**：已有 `ui/message` postMessage → `onMessage` → `App.sendMessage()` 触发新完整链路。但这条路是"绕回顶层"——卡片只能发起全新对话，**无法在同一个 contextId 下携带卡片上下文**。

**扩展设计核心**：需要一个 `a2a-context-propagation` 扩展，让卡片的 `ui/message` 能携带 `contextId + taskId`，让 CF-Agent 在发起 A2A 调用时复用同一个会话上下文，SG-Agent 能感知"这是卡片内的续轮"。

```
卡片 → ui/message + { contextId, taskId }
CF-Agent → A2A message/send + { contextId: 原有 contextId }
SG-Agent → 基于历史 Task 续轮
```

---

### 方向二：异步进展通知

**当前状态**：SSE `/events` 是调用链日志，面向开发者。但**用户侧**没有"任务执行中"的进展推送，SG-Agent 的长时计算（如多轮 LLM + 多次 API 调用）对用户完全黑盒。

**扩展设计核心**：`a2a-task-progress` 扩展。SG-Agent 在 AgentCard 声明支持 `streaming: true`，通过 `message/stream` 返回 `TaskStatusUpdateEvent`，在 `working` 状态时携带进度消息。CF-Frontend 渲染 **skeleton 卡片 + 进度条**。

---

### 方向三：多卡片布局

**当前状态**：一次 A2A 响应只能携带一个 `mcp_ui_resource` data part（虽然 parts[] 支持多个，但 CF-Agent 解析时只取第一个）。

**扩展设计核心**：`a2a-multi-card` 扩展。允许 parts[] 中携带多个 `mcp_ui_resource` data part，每个有独立 `resourceUri` 和 `layout` 字段（`grid` / `tabs` / `carousel`），CF-Frontend 按 `layout` 策略排列渲染。

```json
"data": { "kind": "mcp_ui_resource_collection", "layout": "grid", "cards": [...] }
```

---

### 方向四：交互表单卡片

**当前状态**：卡片只能展示数据，最多通过 `tools/call` 刷新。**没有标准的"用户填写表单 → 提交 → Agent 处理"链路**。

**扩展设计核心**：`a2a-form-action` 扩展。卡片发出 `ui/form/submit` postMessage，携带结构化 form data，CF-Agent 识别后调用 SG-Agent 的特定 skill，结果再注入回同一张卡片或替换为新卡片。

---

## 三个整体方案

这四个方向可以组合成不同的**实施策略**：

**方案 A：各自独立扩展**
每个能力定义独立 URI，AgentCard 分开声明。简单清晰，但 CF-Agent 需要维护多个 Header 和多套解析逻辑。

**方案 B：能力集扩展（Capability Bundle）**
定义一个 `mcp-ui-capabilities/v2` 扩展，包含所有增强能力的 feature flag，一次声明，CF-Agent 按需激活。

**方案 C：分层扩展（推荐）**
- **L1（传输层）**：保持现有 `mcp_ui_resource` 不变
- **L2（交互层）**：新增 `mcp-ui-interaction/v1`，含多轮对话 + 表单
- **L3（布局层）**：新增 `mcp-ui-layout/v1`，含多卡片布局
- **L4（推送层）**：复用 A2A 标准 `streaming` 能力，不新增扩展

分层的好处是**职责分离**，CF-Agent 可以只激活自己需要的层。
