## 二、claude-code 的核心优势维度

### 2.1 工程化基础设施

**claude-code 做到了，huaqi-growing 缺失：**

1. **完整 CI/CD 流水线** — 自动化 lint、typecheck、test、build
2. **Biome 统一格式化+Lint** — 替代 ESLint + Prettier，一个工具两用
3. **死代码消除** — 通过编译时特性标志，禁用功能的代码完全不进入产物
4. **配置迁移脚本体系** — `src/migrations/`，每次配置格式升级有专属迁移脚本，防止用户数据损坏

### 2.2 权限系统设计

**claude-code 拥有细粒度、用户可控的权限系统：**

```
4 种权限模式:
- default   → 逐个提示确认（安全）
- plan      → 规划后批量确认（效率）
- auto      → 自动执行（无人值守）
- bypass    → 企业环境绕过（DevOps）
```

**huaqi-growing 的现状**：依赖 LangGraph Interrupt，只有「有确认/无确认」两种粒度。

**可借鉴**：引入多级权限模式，用户可配置敏感操作（人格更新、内容发布、数据删除）的确认策略。

### 2.3 上下文压缩策略

claude-code 有完整的 `services/compact/` 子系统，包含：
- **微压缩**（单个工具调用结果总结）
- **宏压缩**（整个对话轮次压缩）
- **自动压缩**（token 超预算时触发）

**huaqi-growing 的现状**：无主动压缩。随着对话历史增长，token 成本会线性上涨。

**可借鉴**：实现 `AutoCompact` 策略——当会话 token 超过阈值，自动触发 AI 总结旧消息，保留语义核心，丢弃冗余细节。

### 2.4 CLI 命令系统的模块化

**claude-code 的命令组织：**
```
src/commands/
├── commit/index.ts
├── review/index.ts
├── compact/index.ts
...（80+ 命令，每个独立目录）

// 统一注册
export default buildCommand({
  name: 'compact',
  description: '...',
  execute: async (ctx) => { ... }
})
```

**huaqi-growing 的现状**：Typer 命令直接堆叠在 `cli/commands/` 下，扩展时容易形成巨型文件。

**可借鉴**：引入 `CommandRegistry` 模式——每个命令自描述（name、description、execute），统一注册，方便 `/help` 自动生成和动态加载。

### 2.5 错误处理分层

**claude-code 的三层错误架构：**
```
API 层   → HTTP 状态码分类 + 指数退避重试（withRetry.ts）
工具层   → 权限拒绝 / 执行错误 / 超时
应用层   → 用户友好错误消息 + Sentry 捕获
```

**huaqi-growing 的现状**：依赖 Python 通用异常，无统一异常类体系。

**可借鉴**：
```python
# 引入统一异常类层级
class HuaqiError(Exception): ...
class LLMError(HuaqiError): ...
class MemoryError(HuaqiError): ...
class PipelineError(HuaqiError): ...
class PermissionError(HuaqiError): ...
```

### 2.6 组件化 UI 设计（Ink 架构思想）

**claude-code 的 UI 组件体系（React + Ink）：**
- 每个 UI 元素是独立 React 组件
- `StatusLine`, `TokenWarning`, `MemoryUsageIndicator` 等专注单一职责
- `ThemeProvider` + `ThemedText` 统一主题，一个变量影响所有组件

**huaqi-growing 的现状**：Rich 输出是命令式的（print + Panel），难以主题化和复用。

**可借鉴**：即使不引入 React，也可以：
1. 把反复出现的 UI 片段封装成函数组件（`def render_status_bar()`, `def render_memory_usage()`）
2. 建立颜色/样式的统一常量，而不是每处硬编码 Rich 颜色字符串

### 2.7 启动时并行优化

**claude-code 的启动优化：**
```typescript
// 并行发起所有慢操作
startMdmRawRead()          // 不阻塞
startKeychainPrefetch()    // 不阻塞
// ... 做其他初始化 ...
await ensureKeychainPrefetchCompleted()  // 最后收集结果
```

**huaqi-growing 的现状**：配置加载、模型初始化、向量库连接是串行的，首次启动较慢。

**可借鉴**：用 `asyncio.gather()` 并行初始化独立组件：
```python
config, llm, chroma = await asyncio.gather(
    load_config_async(),
    init_llm_async(),
    init_chroma_async()
)
```

### 2.8 诊断命令（`/doctor` 模式）

**claude-code 有 `/doctor` 命令**：一键检查运行环境、API 连通性、配置合法性。

**huaqi-growing 的现状**：用户遇到问题只能自己排查。

**可借鉴**：`huaqi doctor` 命令，检查：
- Python 版本 / 依赖版本是否满足
- config.yaml 格式是否合法
- LLM API Key 是否有效（发一个测试请求）
- Chroma 向量库是否可用
- 数据目录权限是否正确

### 2.9 使用成本追踪（`/cost` 命令）

**claude-code 精确追踪每次对话的 token 消耗和费用**，并有 `TokenWarning` 组件在接近上限时预警。

**huaqi-growing 的现状**：无 token 成本记录。用户不知道每天消耗了多少，用了多少钱。

**可借鉴**：在每次 LLM 调用后记录 `{prompt_tokens, completion_tokens, model, cost_usd}` 到 `events.db`，`huaqi cost` 命令输出日/周/月用量报表。

### 2.10 记忆系统的「相关性检索」设计

**claude-code 的记忆系统（`src/memdir/`）：**
- `findRelevantMemories.ts` — 相关性匹配
- `memoryAge.ts` — 记忆时效衰减
- 多作者团队记忆支持

huaqi-growing 的混合检索已经很好，但缺少「**记忆时效衰减**」的细粒度控制（已有基础实现，但未精细化）和「**记忆重要性评分**」的自动维护。

## 二、记忆写入机制对比

## 三、记忆检索机制对比

### claude-code：LLM 驱动选择

```
用户输入 → 提取最后真实用户消息
    ↓
startRelevantMemoryPrefetch()（非阻塞并发）
    ↓
scanMemoryFiles() → 按 mtime 扫描最多 200 个文件
    ↓
Sonnet 评估相关性（JSON Schema 输出）
    → 理解上下文语义，不只是关键词
    ↓
返回 top-5，注入用户消息前
```

### huaqi-growing：算法驱动选择

```
用户消息进入 retrieve_memories 节点
    ↓
L3 向量库混合检索：
    BM25 召回 top-20 → 归一化评分
    向量召回 top-20 → cosine 相似度
    融合：0.7×向量 + 0.3×BM25 + 0.1×时间衰减
    → top-3
    ↓
L2 今日文件扫描：
    提取 2-gram 关键词 → 遍历最近 20 文件
    → snippet 抽取
    ↓
去重 + 1000 token 截断 → top-5
    ↓
注入 system_prompt
```

**关键差异**：

| | claude-code | huaqi-growing |
|---|---|---|
| **判断方式** | Sonnet 语义理解 | 算法数学评分 |
| **优势** | 理解隐含意图，如"最近情绪" | 确定性、低延迟、无额外 LLM 调用 |
| **劣势** | 每次检索多消耗 token | 无法理解隐含关联 |
| **召回上限** | 200 文件扫描 | BM25 + 向量各 top-20 |

### 改进4：存储重复检测（防止知识库膨胀）

**问题**：同一个用户习惯，可能被多次对话反复提炼，存成多条重复记忆。

**具体做法**：在写入新知识前，先检查知识索引里有没有 `description` 相似的条目；相似度用 BM25（不需要向量，已有`TextSearch`），超过阈值则更新现有文件而不是新建。

---

**不建议做的**：
- 照搬 `extractMemories` 的完整子 Agent 方案（成本高、复杂度高，huaqi 场景不需要这个量级）
- 引入团队记忆（huaqi 是个人产品）
- 把现有对话归档改成 claude-code 的格式（历史数据迁移成本大，且对话归档本身没问题）

**最终判断**：huaqi 的检索基础设施（BM25 + 向量 + 时间衰减）比 claude-code 更扎实，真正的差距在**写入层**——存的东西质量不够高，导致检索出来的内容对 LLM 的帮助有限。

## 问题 1：存的是原始对话，不是知识（最核心）
[2026-04-01 14:42:34] [assistant]: **问题根源**：`save_conversation` 节点把每次对话的所有 turns 原封不动存成 Markdown 文件，同时把每个 turn 写入 Chroma。一次「今天吃什么」的对话和一次「我打算明年换工作」的对话，在存储层被一视同仁。

**改法**：新增一个 `knowledge/` 目录专门存结构化知识，在 `save_conversation` 之后异步触发知识提炼。

**新建文件：`huaqi_src/memory/knowledge_store.py`**

```python
"""结构化知识存储

对话后异步提炼的长期知识，与原始对话归档分离。
"""

from dataclasses import dataclass, field
from datetime import datetime
from pathlib import Path
from typing import Any, Dict, List, Literal, Optional
import re
import yaml


KnowledgeType = Literal["preference", "habit", "goal", "fact"]


@dataclass
class KnowledgeItem:
    name: str
    description: str
    knowledge_type: KnowledgeType
    content: str
    source_date: str
    created_at: str = field(default_factory=lambda: datetime.now().isoformat())
    updated_at: str = field(default_factory=lambda: datetime.now().isoformat())


class KnowledgeStore:
    """结构化知识存储

    目录结构：
        memory/knowledge/
        ├── INDEX.md              ← 全局索引（加载进系统提示）
        ├── pref_morning_work.md  ← 单条知识文件
        └── habit_running.md
    """

    INDEX_FILENAME = "INDEX.md"
    MAX_INDEX_LINES = 100

    def __init__(self, memory_dir: Path):
        self.knowledge_dir = Path(memory_dir) / "knowledge"
        self.knowledge_dir.mkdir(parents=True, exist_ok=True)

    @property
    def index_path(self) -> Path:
        return self.knowledge_dir / self.INDEX_FILENAME

    def _slug(self, name: str) -> str:
        """将名称转为文件名"""
        s = re.sub(r"[^\w\u4e00-\u9fff]", "_", name.lower())
        return s[:40]

    def save(self, item: KnowledgeItem) -> Path:
        """保存一条知识，自动更新索引"""
        filename = f"{item.knowledge_type[:4]}_{self._slug(item.name)}.md"
        filepath = self.knowledge_dir / filename

        content = (
            f"---\n"
            f"name: {item.name}\n"
            f"description: {item.description}\n"
            f"type: {item.knowledge_type}\n"
            f"source_date: {item.source_date}\n"
            f"updated_at: {item.updated_at}\n"
            f"---\n\n"
            f"{item.content}\n"
        )
        filepath.write_text(content, encoding="utf-8")
        self._update_index(item, filename)
        return filepath

    def _update_index(self, item: KnowledgeItem, filename: str):
        """更新 INDEX.md，若同名条目已存在则替换"""
        existing_lines: List[str] = []
        if self.index_path.exists():
            existing_lines = self.index_path.read_text(encoding="utf-8").splitlines()

        new_line = f"- [{item.name}]({filename}) — {item.description} ({item.source_date})"

        # 找到同文件名的旧条目并替换，否则追加
        replaced = False
        for i, line in enumerate(existing_lines):
            if f"({filename})" in line:
                existing_lines[i] = new_line
                replaced = True
                break

        if not replaced:
            # 按 type 分组插入（保持索引可读性）
            section_header = {
                "preference": "## 偏好",
                "habit": "## 习惯",
                "goal": "## 目标",
                "fact": "## 关于用户",
            }.get(item.knowledge_type, "## 其他")

            if section_header not in existing_lines:
                existing_lines += ["", section_header]
            insert_at = existing_lines.index(section_header) + 1
            existing_lines.insert(insert_at, new_line)

        # 截断超长索引（防止系统提示膨胀）
        if len(existing_lines) > self.MAX_INDEX_LINES:
            existing_lines = existing_lines[-self.MAX_INDEX_LINES:]

        self.index_path.write_text("\n".join(existing_lines) + "\n", encoding="utf-8")

    def load_index(self) -> str:
        """加载索引内容（用于注入系统提示）"""
        if not self.index_path.exists():
            return ""
        return self.index_path.read_text(encoding="utf-8")

    def find_similar(self, description: str, threshold: float = 0.5) -> Optional[str]:
        """用 BM25 找相似知识文件名，用于去重判断"""
        from .search.text_search import TextSearch
        if not self.index_path.exists():
            return None

        lines = [
            l for l in self.index_path.read_text(encoding="utf-8").splitlines()
            if l.startswith("- [")
        ]
        if not lines:
            return None

        ts = TextSearch(algorithm="bm25")
        for line in lines:
            ts.add_document(line)

        results = ts.search(description, top_k=1)
        if results and results[0].score > threshold:
            # 从行里解析文件名
            m = re.search(r"\(([^)]+\.md)\)", results[0].content)
            return m.group(1) if m else None
        return None
```

```python
            user_msg = turn["user_message"]
            assistant_msg = turn["assistant_response"]

            # 过滤规则：跳过低价值 turn
            combined = f"{user_msg}\n{assistant_msg}"
            if not _is_worth_indexing(user_msg, combined):
                continue

            content = f"用户：{user_msg}\nHuaqi：{assistant_msg}"
            doc_id = f"{session_id}_turn_{i}"

            embedding = embedder.encode(content)
            chroma.add(
                doc_id=doc_id,
                content=content,
                metadata={
                    "session_id": session_id,
                    "turn_index": i,
                    "date": timestamp.strftime("%Y-%m-%d"),
                    "type": "conversation",
                },
                embedding=embedding.tolist() if hasattr(embedding, "tolist") else list(embedding),
            )
    except Exception:
        pass


def _is_worth_indexing(user_msg: str, combined: str) -> bool:
    """判断这个 turn 是否值得写入 Chroma

    规则：
    - 合并内容太短（<30字）：闲聊/打招呼，跳过
    - 纯问候词：跳过
    - 包含实质性内容关键词：优先保留
    """
    if len(combined.strip()) < 30:
        return False

    GREETINGS = {"你好", "hi", "hello", "嗨", "在吗", "早", "晚安", "再见", "拜拜", "谢谢", "好的", "ok", "好"}
    user_lower = user_msg.strip().lower()
    if user_lower in GREETINGS or len(user_lower) < 4:
        return False

    return True
```

## 问题 5：检索结果截断 200 字符，丢失上下文
[2026-04-01 14:44:13] [assistant]: **问题根源**：`generate_response` 里把每条记忆截断到 200 字符（第 267 行），然后总体截断到 1000 字符。对于包含多轮对话的 Chroma 记录，200 字符通常只到对话的一半。

**改法**：两处截断值调大，同时把知识库记忆（来自 `knowledge/` 目录）和对话记忆分开处理，前者不需要截断（本来就是精炼的）：

```python
# generate_response 节点中的注入逻辑

    # knowledge 类记忆（来自 KnowledgeStore，较短且精炼）
    knowledge_items = [m for m in memories if m.get("source") == "knowledge"]
    # 对话类记忆（来自 Chroma，可能较长）
    conversation_items = [m for m in memories if m.get("source") != "knowledge"]

    parts = []
    if knowledge_items:
        k_lines = [_format_memory_with_age(m) for m in knowledge_items]
        parts.append("【长期知识】\n" + "\n".join(k_lines))

    if conversation_items:
        # 对话记忆截断到 400 字符（原来是 200）
        c_lines = []
        for m in conversation_items:
            m_copy = dict(m)
            m_copy["content"] = m_copy.get("content", "")[:400]
            c_lines.append(_format_memory_with_age(m_copy))
        parts.append("【相关对话记忆】\n" + "\n".join(c_lines))

    combined = "\n\n".join(parts)
    if len(combined) > 2000:   # 总上限从 1000 调到 2000
        combined = combined[:2000] + "\n...(截断)"

    system_prompt += f"\n\n{combined}"
```
