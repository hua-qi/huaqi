## 领用审批流深度分析

### 一、三套审批流并存

这是这个模块最核心的复杂点——不是一套审批流，而是**三套流根据预算类型分支**：

```
领用申请
    ├── 项目预算流   →  走项目预算 BPF 节点配置
    ├── 活动预算流   →  走活动预算 BPF 节点配置
    └── 一次性预算流 →  业务员发起 → 上级+预算BP并签 → 知会核算
                        → 后端调预算接口创建预算申请
                        → 预算单号回写到领用单
```

项目/活动预算是**预算系统已有预算行**，领用时占用已有额度；一次性预算是**边申请边创建预算**，流程里才生成预算单。

---

### 二、状态与库存的双向联动

```
提交申请
  ↓
【占用】预算额度 + 公共库存（此时库存现有量未减少）
  ↓
库管员审批节点（最关键节点）
  - 分配具体子库存
  - 若多子库存 → 必须手动选择，且做库存现有量 ≥ 领用数量校验
  - 若单子库存 → 自动带出
  ↓
审批通过 → 实际出库
  ↓
【释放】预算占用（转为实际扣减）+ 子库存现有量减少
```

这里有一个**时间差设计**值得关注：占用和实际扣减是两个时间点，意味着并发领用场景下库存可能被超占，依赖提交时的校验兜底。

---

### 三、驳回/撤回重提的复杂性

- 被驳回或撤回后，单据回到草稿态，可编辑重提
- 重提时需要**重新触发预算占用和库存占用**
- 这意味着之前的占用必须在撤回/驳回时及时释放，否则会双占

这块是最容易出 bug 的地方，状态机需要非常严谨。

---

### 四、财务节点的校验逻辑

审批链路末端有财务 BP 节点，有一条特殊校验规则：

> 费用类别为**非团建费** + **预算承担部门无成本中心** → 财务 BP 节点报错，不允许通过

这个规则意味着：领用单提交时不报错，到财务 BP 节点才爆——用户体验上有一定滞后，前置校验应该更早做。

---

### 五、最复杂的工程点总结

| 复杂点 | 原因 |
|---|---|
| 三套审批流分支 | 预算类型不同，BPF 流程 ID 不同，传参结构也不同 |
| 一次性预算回写 | 审批中途调外部预算系统接口，异步回写预算单号 |
| 库管员子库存分配 | 多子库存时动态必填，单子库存自动带出，两套逻辑 |
| 撤回/驳回状态管理 | 预算和库存占用的释放时机必须精确，否则重复占用 |
| 并发超占 | 占用和扣减两个时间点，高并发场景需额外保护 |

## 天舟如何解决预算和库存占用问题

先说结论：**文档层面设计是清晰的，但并发保护主要靠校验而非锁，是一个"乐观并发"策略。**

---

### 一、库存占用的设计

天舟把库存分成三个字段来管理：

```
总库存 = 占用库存 + 可用库存

提交领用申请时：可用库存 → 转入占用库存（现有量不变）
审批通过出库后：占用库存减少 + 现有量减少
撤回/驳回时：  占用库存 → 回到可用库存
```

提交时的校验逻辑（四期）比较有意思，不是简单校验总库存，而是：

> **通用库存量 + 领用类型子仓库现有量 - 已申请但未分配子仓库的数量 ≥ 领用数量**

这个公式本质上是在做"软占用"，把审批中的单据也计入已消耗，防止多人同时提交把库存超占。

**但这里有一个明显的并发窗口**：两个人同时通过校验、同时提交，都能成功占用——文档里没有提到乐观锁或数据库行锁的机制，**是依赖 Kaleido 平台的事务能力兜底的**，这块是潜在风险点。

---

### 二、预算占用的设计

预算和库存是**同一个提交动作触发两个调用**，但本质不同：

| | 库存占用 | 预算占用 |
|---|---|---|
| 存储位置 | 天舟自己的数据库 | 外部预算系统 |
| 占用方式 | 字段更新（占用库存字段+1） | 调接口，occupationAmount 正数 |
| 释放方式 | 字段回滚 | 再调接口，occupationAmount **负数** |
| 幂等性 | 依赖平台事务 | 依赖预算系统接口 |

释放预算用的是"负数金额再次调用占用接口"，不是单独的释放接口——这意味着**如果提交预算占用成功、但后续某步失败，必须有补偿机制主动调负数释放**，否则预算会永久被占用。

---

### 三、撤回/驳回的状态机

```
草稿
  ↓ 提交
审批中  ──────────────→ 占用预算(外部接口) + 占用库存(本地)
  │
  ├── 撤回（仅业务审批前可撤回）
  │     → 调预算接口 occupationAmount=-X 释放
  │     → 本地库存占用回滚
  │     → 单据回草稿态
  │
  ├── 驳回（任意审批节点）
  │     → 同上，释放预算+库存
  │     → 单据回草稿态，可重新提交
  │
  └── 审批通过
        → 实际出库，现有量减少
        → 预算占用转为实际扣减（由预算系统处理）
```

重提时会**重新走一遍占用**，所以前一次的占用必须在驳回/撤回时已经完整释放，否则就双占了。

---

### 四、最大的隐患在哪

**两个系统的原子性问题**

提交时要同时成功：
1. 本地库存字段更新
2. 外部预算系统接口调用成功

如果 **1 成功、2 失败**，或者 **2 成功、1 失败**，系统就进入不一致状态。文档里没有描述补偿事务或重试机制，这块要么是 Kaleido 平台处理了，要么是实际运行中靠人工对账兜底的。

**这在内部低频系统里是可以接受的工程取舍**——天舟的领用场景并发量很低，不是秒杀场景，出现不一致的概率极低，成本不值得引入分布式事务。

### 两种锁的思路对比

**悲观锁**：假设冲突一定会发生，所以操作前先把数据锁住，别人必须等我改完才能读/写。
> 类比：厕所门锁，进去就把门锁上，别人只能排队等。

**乐观锁**：假设冲突很少发生，不提前加锁，但提交时检查"数据在我读取之后有没有被别人改过"。
> 类比：Google 文档协作，写完提交时发现别人也改了同一行，系统提示冲突让你解决。

---

### 乐观锁的具体实现

数据库里通常加一个 `version` 字段：

```
读取数据：  库存=100, version=5
我准备改成：库存=90

提交时执行：
UPDATE stock SET amount=90, version=6
WHERE id=xxx AND version=5   ← 关键：带上读取时的版本号

如果 version 已经被别人改成 6 了 → 这条 SQL 影响行数为 0 → 报错重试
```

## SKU 在天舟中的状态流转

天舟里 SKU 本身没有单独的"状态字段"，它的状态是通过**库存数量的变化**来体现的，本质上是一个**数量账本**。

---

### 一、SKU 的生命周期

```
商品主数据录入（启用）
        ↓
  可参与入库 / 领用 / 售卖
        ↓
  商品禁用 → 不再出现在领用/售卖选择项中
```

SKU 主数据只有**启用/禁用**两个状态，禁用后不能新建单据，但历史数据仍然存在。

---

### 二、库存视角下的 SKU 数量流转

这才是核心，SKU 的"状态"实际上是它的库存数量在不同桶之间的流动：

```
                    ┌─────────────────────────────┐
                    │       总库存 (现有量)         │
                    │  = 可用库存 + 占用库存        │
                    └─────────────────────────────┘
                           ↑              ↑
               采购入库      │              │  领用申请提交
               盘盈          │              │  → 可用 → 占用
               成本调整入库   │              │
                            │              ↓
                            │        审批通过出库
                            │        售卖出库
                            │        盘亏/报损
                            │        → 占用/可用 → 减少
```

每一类业务动作对应的数量变化：

| 业务动作 | 可用库存 | 占用库存 | 总库存 |
|---|---|---|---|
| 采购入库确认 | +N | — | +N |
| 领用申请提交 | -N | +N | 不变 |
| 领用撤回/驳回 | +N | -N | 不变 |
| 领用审批通过出库 | — | -N | -N |
| 售卖出库 | -N | — | -N |
| 库存盘盈 | +N | — | +N |
| 库存盘亏/报损 | -N | — | -N |
| 成本调整 | 不变 | 不变 | 不变（只调成本金额）|

---

### 三、一个关键设计：盘点不参与本期成本

这是三期里一个值得注意的设计决策：

> 盘盈盘亏产生的数量变动，**不计入本期成本核算，只算在期初剩余数量里**

意思是：
- 本期成本 = 只看采购入库带来的成本
- 盘点调整的数量变化推迟到**下一期的期初**才体现在成本里

这样做的好处是**保护本期成本数据的纯洁性**，避免盘点操作影响已经在走流程的财务入账。代价是盘点和成本之间有一期的滞后。

---

### 四、SKU 映射的独立维度

SKU 还有一个平行的映射体系，和库存数量流转是独立的：

```
天舟 SKU（内部编码）
    ├── 快手小店渠道编码 A
    ├── 快手小店渠道编码 B   ← 四期重构：支持 1:N 映射
    └── 银豹渠道编码 C
```

四期之前是固定字段列（一个 SKU 只能映射一个渠道编码），四期重构成了独立映射表，才支持同一个 SKU 对应多个渠道编码。这个改动主要影响的是售卖导入时的编码匹配逻辑。

### 核心思想

不追踪每批货物的具体成本，而是**每个会计期间结束时，用这一期的总成本 ÷ 总数量，算出一个平均单价**，作为本期所有出库的成本。

---

### 一个直观的例子

```
某 SKU，本期发生：

期初结余：100个，单价 10元  → 成本 1000元
本期入库：50个，  单价 14元  → 成本  700元
─────────────────────────────────────
合计：   150个              → 成本 1700元

PAC 单价 = 1700 ÷ 150 = 11.33元

本期出库（领用/售卖）100个
→ 出库成本 = 100 × 11.33 = 1133元

期末结余：50个，单价 11.33元，成本 567元
```

---

### 为什么天舟选 PAC 而不是其他方法

常见成本法对比：

| 方法 | 思路 | 适用场景 |
|---|---|---|
| **PAC 期间平均** | 期末统一算平均价 | 批次多、价格波动大，内部管理用 |
| FIFO 先进先出 | 先买的先出 | 需要精确追踪批次 |
| 移动加权平均 | 每次入库后立即重算平均价 | 实时性要求高 |
| 个别计价 | 每件货单独追踪 | 高值商品如珠宝 |

天舟是**内部礼品管理**，商品批次多、采购价每期都可能不同，不需要追踪具体批次，PAC 是最合适且最简单的选择。

---

### 和库存期间的关系

这也是为什么天舟要有**库存期间开关**的原因——PAC 必须在一个期间结束时才能"算出"这期的平均成本，期间关闭前出库成本是暂估的，**关闭后才锁定最终成本**，再拿去财务入账。

## SKU 状态流转与财务的关系

核心结论：**SKU 的数量流转产生成本数据，成本数据经过期间锁定后，才进入财务体系。**

财务不关心单次流转，只关心**期间汇总结果**。

---

### 一、哪些流转会产生财务影响

```
数量变动类型          是否影响财务成本
─────────────────────────────────
采购入库              ✅ 带入含税/不含税成本
领用出库（审批通过）    ✅ 按 PAC 单价扣减成本，生成领用入账明细
售卖出库              ✅ 按 PAC 单价扣减成本
盘盈盘亏/报损         ⚠️  不计入本期成本，推迟到下期期初
成本调整              ✅ 直接修正成本金额，走独立审批流
领用占用（审批中）     ❌ 只锁库存，不产生财务数据
```

---

### 二、从流转到财务的完整链路

```
SKU 数量流转
    ↓
库存期间关闭
    → 触发 PAC 计算，锁定本期平均单价
    ↓
两条财务出口：

出口1：领用出库
  → 生成「领用入账明细」
  → 财务 BP 在「领用入账确认」界面逐条确认
  → 带出：费用类别、五段值、成本中心、领用主体
  → 确认后推送财务系统做凭证

出口2：售卖出库
  → 成本数据可导出
  → 目前是离线对接财务系统，非系统内自动生成凭证
```

---

### 三、期间关闭是财务的分水岭

```
期间开启中                    期间关闭后
─────────────────────────────────────────
数量可以自由流转               数量流转已锁定
成本是暂估值                   PAC 单价已锁定
领用入账明细可能不完整          可以正式财务入账
```

这就是为什么财务流程强依赖库存期间——**不关期间就没有准确的成本，没有成本就无法入账。**

```
10月：
  采购入库 100个 @10元
  领用出库  60个（审批通过）→ 占用入账明细，成本暂估
  售卖出库  20个

10月31日：库存期间关闭
  PAC = (100×10) ÷ 100 = 10元/个
  领用出库成本 = 60 × 10 = 600元（锁定）
  售卖出库成本 = 20 × 10 = 200元（锁定）

11月：

  核对 600元，填写成本中心/费用类别
  确认入账 → 推送财务系统
```

## 天舟前端性能优化

文档里明确提到的优化点只有两个，但背后的思路值得深聊。

---

### 一、报表大数据量问题 —— 异步计算 + 轮询 + KIM 推送

这是最核心的优化，解决的问题是：

> 库存余额表、销售情况表这类报表，数据量大，后端计算耗时长，用户点击后如果同步等待会直接超时或白屏。

天舟的解法是一套**异步任务机制**：

```
用户点击「导出/生成报表」
        ↓
前端发起请求 → 后端返回「任务ID」，立即返回（不等计算结果）
        ↓
前端开始轮询任务状态（每隔 N 秒请求一次）
        ↓
后端计算完成 → 通过 KIM（快手内部 IM）推送消息通知用户
        ↓
用户收到消息 → 点击链接 → 前端展示/下载结果
```

轮询 + 推送是双保险：
- **轮询**：用户停留在页面时能感知进度
- **KIM 推送**：用户离开页面后也能被通知到，不会错过结果

这个模式本质上是把**同步接口改造成异步任务队列**，是大数据量场景的标准解法。

---

### 二、看板分块加载

看板页面有多个卡片/图表，如果等所有数据全部返回再渲染，首屏时间很长。

解法是**每个卡片独立请求、独立渲染**：

```
❌ 之前：一个接口返回所有看板数据 → 全部到了才渲染
✅ 优化：每个卡片独立请求 → 谁先回来谁先渲染
```

用户感知上是"页面在逐步填充"，比"白屏等待后一次性出来"体验好很多。

---

### 三、移动端适配

天舟是内部系统，主要在 PC 用，但五期做了移动端兼容。

策略比较务实，没有做完整的响应式布局，而是：

> 设置页面最小宽度，设备宽度不够时页面可以横向滚动

这是典型的**以最小成本换最大兼容性**的工程取舍——毕竟内部系统移动端用户极少，不值得投入完整的响应式改造。

### 四、低代码平台下的隐性优化

天舟基于 Kaleido，有一个特殊的性能约束值得提：

**平台本身的能力边界就是性能边界**，很多通用优化手段（虚拟滚动、懒加载、代码分割）在低代码平台里不能自由实现，只能在平台允许的范围内做优化。

- 超出平台能力的场景用架构绕过（异步任务代替同步请求）

### 二、性能问题的根因

在 Kaleido 低代码平台里，性能瓶颈集中在**后端流（Flow）的执行耗时**上，而不是传统意义上的前端渲染问题。原因是：

> Kaleido 的业务逻辑写在"流"里，流默认用 **JS 引擎执行**，JS 引擎比 Java 慢得多

---

### 三、核心优化手段：JS 转 Java

这是整个方案最重要的一条：

```
原来：Flow 节点用 JS 编写 → JS 引擎执行 → 慢
优化：Flow 节点改用 Java 编写 → JVM 执行 → 快
```

不同情况的处理策略：

| 情况 | 策略 |
|---|---|
| 普通业务流 | 直接 JS → Java 改写 |
| 平台回调流（不能直接改） | 改造回调流本身，尽量转 Java |
| 平台内置流（完全不能改） | 其余节点转 Java，内置节点找平台侧支持 |
| 外部系统 OpenAPI 调用慢 | 其余节点转 Java，外部系统配合改造 |
| 流间参数过大 | 缩减传参，防止 OOM |

---

### 四、增量改造原则

一个值得关注的工程策略：

> **不修改原有流，而是复制一个新流再改造**

好处是：
- 线上原有逻辑不受影响，零风险
- 新流可以独立验证、灰度切换
- 出问题可以随时回滚到旧流

---

### 五、结合天舟来看

两个项目都在 Kaleido 上，但优化思路有差异：

| | 天舟进销存 | HR 服务台 |
|---|---|---|
| 性能瓶颈 | 大数据量报表的用户等待 | 接口流执行耗时（JS引擎慢） |
| 优化手段 | 异步任务 + 轮询 + KIM 推送 | JS 流 → Java 流改写 |
| 前端层面 | 分块加载、移动端适配 | 指标监控体系建立 |
| 本质 | 绕过等待问题（体验优化） | 直接提速（性能优化） |

两种思路代表了低代码平台下性能优化的两个方向：一个是**改变交互模式让用户感知不到慢**，另一个是**直接让执行变快**。
