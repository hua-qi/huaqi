- **Pipeline 执行**：解析并运行 YAML 定义的流水线（stages/steps）

- **Multimodal**：多模态（文本/图像/音频/视频）
- **Reasoning Model**：推理模型（如 o1/DeepSeek-R1），显式 CoT
- **小模型 / 端侧模型**：SLM（如 Phi、Qwen 系列）

**新范式**
- **Vibe Coding**：自然语言驱动编程，AI 生成全部代码
- **Computer Use**：AI 直接操控 GUI/浏览器
- **Memory**：给 Agent 加长期记忆
[2026-04-01 21:04:12] [user]: Agentic Workflow 是什么都东西
[2026-04-01 21:04:22] [assistant]: **Agentic Workflow** 是一种以 AI Agent 为核心驱动的任务执行模式，区别于传统的"单次问答"交互。

**核心思想：让 AI 像人一样迭代工作**

传统 LLM 调用：
```
用户输入 → LLM 一次性输出 → 结束
```

Agentic Workflow：
```
目标 → 规划 → 执行 → 观察结果 → 反思 → 再规划 → ... → 完成
```

**四个关键能力（吴恩达总结）**

| 能力 | 说明 |
|------|------|
| **Reflection** | 自我反思、检查输出质量并修正 |
| **Tool Use** | 调用搜索、代码执行、API 等外部工具 |
| **Planning** | 将复杂目标拆解为子任务并排序 |
| **Multi-Agent** | 多个专职 Agent 协作分工 |

**典型例子**

写一篇报告：
1. Agent 先搜索资料
2. 写初稿
3. 自我 Review 发现问题
4. 修改润色
5. 输出最终版

**代表框架**：LangGraph、AutoGen、CrewAI、Dify、Coze

本质上就是：**让 AI 从"回答问题"变成"完成任务"**，具备自主循环执行的能力。
[2026-04-01 21:05:56] [user]: Agentic workflow 和 multi model 之间的区别
[2026-04-01 21:06:04] [assistant]: 这两个是**完全不同维度**的概念，不存在直接对比关系：

| | **Agentic Workflow** | **Multimodal** |
|---|---|---|
| **是什么** | 任务执行**模式** | 模型**能力** |
| **关注点** | AI 如何完成任务（规划、循环、工具调用） | AI 能处理哪些数据类型（文本/图像/音频/视频） |
| **解决问题** | 复杂多步骤任务的自主完成 | 跨模态理解与生成 |

**类比**：
- Multimodal = 一个人会看图、听声音、读文字（**感知能力**）
- Agentic Workflow = 这个人能自主规划、反思、协作完成一个项目（**工作方式**）

两者可以结合：一个 Agentic Workflow 中的 Agent，可以调用 Multimodal 模型来处理图片，这是互补关系而非竞争关系。
[2026-04-01 21:06:50] [user]: Agentic workflow 和 multi agent 之间的区别
[2026-04-01 21:06:58] [assistant]: 两者是**包含关系**，不是并列关系：

**Multi-Agent 是 Agentic Workflow 的一种实现模式**

| | **Agentic Workflow** | **Multi-Agent** |
|---|---|---|
| **是什么** | 任务执行的**整体框架/模式** | Agentic Workflow 中的**一种协作机制** |
| **范围** | 更大，包含单 Agent 和多 Agent | 更具体，特指多个 Agent 协作 |
| **关注点** | AI 如何自主循环完成任务 | 多个专职 Agent 如何分工协同 |

**关系图：**
```
Agentic Workflow
├── 单 Agent（规划 + 工具调用 + 反思）
└── Multi-Agent（多个 Agent 协作）
    ├── Agent A（负责搜索）
    ├── Agent B（负责写作）
    └── Agent C（负责审查）
```

**简单说：**
- Agentic Workflow = "自主完成任务"的工作范式
- Multi-Agent = "多人分工协作"的组织方式，是 Agentic Workflow 的子集
[2026-04-01 21:08:56] [user]: agent 的定义
[2026-04-01 21:09:04] [assistant]: **Agent（智能体）** 的核心定义：

> 能够**感知环境、自主决策、采取行动**以完成目标的 AI 系统

**最简模型：**
```
感知（Perception）→ 思考（Reasoning）→ 行动（Action）→ 循环
```

**与普通 LLM 的本质区别：**

| | **LLM** | **Agent** |
|---|---|---|
| 交互方式 | 被动响应 | 主动执行 |
| 生命周期 | 单次调用 | 持续循环 |
| 工具使用 | 无 | 可调用外部工具 |
| 目标导向 | 回答问题 | 完成任务 |

**构成要素：**
- **Brain（大脑）**：LLM，负责推理决策
- **Memory（记忆）**：短期（上下文）+ 长期（向量数据库）
- **Tools（工具）**：搜索、代码执行、API 调用等
- **Action（行动）**：执行决策结果

**一句话：** LLM 是"问答机器"，Agent 是"能干活的员工"——给它一个目标，它自己想办法完成。
[2026-04-01 21:12:48] [user]: ReAct 模型就是 Agent 的模型咯？
[2026-04-01 21:12:57] [assistant]: 不完全准确。

**ReAct 是 Agent 的一种经典实现框架**，不是唯一的。

**ReAct = Reasoning + Acting**
```
思考（Thought）→ 行动（Action）→ 观察（Observation）→ 思考 → ...
```

**Agent 的常见实现框架：**

| 框架 | 特点 |
|---|---|
| **ReAct** | 交替推理+行动，最经典 |
| **CoT**（Chain of Thought） | 只推理，不行动 |
| **ReWOO** | 先一次性规划所有步骤，再执行 |
| **Reflexion** | 加入自我反思和记忆 |
| **Plan-and-Execute** | 先完整规划，再逐步执行 |

**关系：**
```
Agent（概念）
└── 需要一个决策框架来实现
    ├── ReAct（最常用）
    ├── Reflexion
    ├── Plan-and-Execute
    └── ...
```

**一句话：** ReAct 是目前最流行的 Agent 推理框架，但 Agent 不等于 ReAct。

[2026-04-02 11:43:25] [assistant]: **BFF = Backend For Frontend**

专门为前端服务的后端层，介于前端和真正的后端服务之间。

```
前端 → BFF（Node.js）→ 后端微服务A
                    → 后端微服务B
```

- 后端返回的数据格式不适合前端，BFF 做裁剪/聚合
- 前端一个页面需要调多个接口，BFF 合并成一个请求
- 接口鉴权、数据转换逻辑不用堆在前端

**对你的意义：** 写 BFF 是前端转全栈最平滑的路径，用 Node.js 写，语言没有切换成本。
