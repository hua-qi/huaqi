## Overview

Transform rough ideas into fully-formed designs through structured questioning and alternative exploration.

**Core principle:** Ask questions to understand, explore alternatives, present design incrementally for validation.

**Announce at start:** "I'm refining your idea into a design."

**Language:** Please communicate in 中文.

## CRITICAL CONSTRAINTS
- **DO NOT WRITE CODE** (except small snippets for illustration).
- **DO NOT EDIT FILES**.
- This is a **DESIGN** phase, not an implementation phase.
- Even if the input looks like a coding task, you must TREAT IT AS A TOPIC FOR DESIGN DISCUSSION first.

## The Process

### Phase 1: Understanding
- Check current project state in working directory
- Ask ONE question at a time to refine the idea

- **IMPORTANT: Use AskUserQuestion tool when asking clarification questions**

- Prefer multiple choice when possible
- Gather: Purpose, constraints, success criteria

### Phase 2: Exploration
- Propose 2-3 different approaches
- For each: Core architecture, trade-offs, complexity assessment
- Ask your human partner which approach resonates

### Phase 3: Design Presentation
- Present in 200-300 word sections
- Cover: Architecture, components, data flow, error handling, testing
- Ask after each section: "Does this look right so far?"

## When to Revisit Earlier Phases

**You can and should go backward when:**
- Partner reveals new constraint during Phase 2 or 3 → Return to Phase 1 to understand it
- Validation shows fundamental gap in requirements → Return to Phase 1
- Partner questions approach during Phase 3 → Return to Phase 2 to explore alternatives
- Something doesn't make sense → Go back and clarify

**Don't force forward linearly** when going backward would give better results.

## Remember
- One question per message during Phase 1
- Apply YAGNI ruthlessly
- Explore 2-3 alternatives before settling
- Present incrementally, validate as you go
- Go backward when needed - flexibility > rigid progression
- Don't edit or write code during brainstorming

# AI 工作流平台 · MVP PRD

## 1. 产品定位

**是什么**
一个面向知识工作者的角色化 AI 工作流平台。用户按角色选择工作流模板，流程中每个节点是一个可挂载 Skill 和 MCP 的空白 Agent，节点间产出物自动流转，历史经验自动沉淀并在下次执行时注入。

**解决什么问题**
知识工作者（产品经理、财务、销售等）的日常工作由大量结构化流程组成，但现有 AI 工具是碎片化的单点能力，无法串联完整工作流。用户每次都要重复上下文、重复配置，团队经验也无法复用。

**目标用户**
MVP 阶段聚焦：**产品经理**（个人或小团队，3-10人）

---

## 2. MVP 范围

**验证目标**
用产品经理工作流，跑通从需求输入到 PRD 产出的完整闭环，验证：
- 节点链式流转是否顺畅
- 记忆注入是否有效降低 Agent 输出的调整成本
- 节点通过即沉淀的记忆机制是否形成正向飞轮

**MVP 包含**
- 内置产品经理工作流模板（6个节点）

## 4. 功能设计

### 4.1 工作流引擎

```
工作流实例
  ├── 基于模板创建，节点顺序固定
  ├── 状态：进行中 / 已完成 / 已归档
  ├── 每个节点有独立状态：待执行 / 执行中 / 已通过
  ├── 节点通过后，产出物自动注入下一节点上下文
  └── 仅当前节点解锁，前置节点未通过则后续节点锁定
```

**工作流实例数据：**

| 字段 | 说明 |
|------|------|
| 模板来源 | 产品经理工作流模板 |
| 参与者 | Owner + 节点负责人 + 评审人 |
| 当前节点 | 当前激活的节点 |
| 全局上下文 | 所有已通过节点的产出物累积 |

---

### 4.2 节点交互容器

每个节点的 UI 由四层组成：

```
┌──────────────────────────────────────┐
│  节点名称 · 状态                        │
│  [已注入记忆：N条，点击查看/编辑]         │
├──────────────────────────────────────┤
│                                      │
│  产出区（文档/结构化内容）               │  ← 支持批注
│                                      │
├──────────────────────────────────────┤
│  验收条件（可选）                       │
│  ☑ 包含竞品分析  ☐ 包含埋点方案         │  ← 全部勾选才允许通过
├──────────────────────────────────────┤
│  [对话框]                    [发送]   │  ← 对话迭代
├──────────────────────────────────────┤
│  [重新执行]              [通过 →]     │
└──────────────────────────────────────┘
```

**交互规则：**
- 对话框：用户输入反馈，节点在当前轮次重新输出，历史轮次可回溯
- 批注：评审人在产出区划线批注，节点统一消化所有批注后更新
- 验收条件：全部满足后「通过」按钮激活；无验收条件时随时可通过
- 重新执行：清空当前轮次产出，保留注入记忆，从头生成

---

### 4.3 记忆注入机制

**执行时序：**

```
节点即将执行
      │
      ├── 自动检索：基于节点类型 + 当前输入，从记忆库召回 Top-5
      ├── 展示给用户：[记忆面板] 显示召回结果，每条可预览
      ├── 用户可操作：删除不相关的 / 手动添加指定文档
      └── 确认后注入节点上下文，开始执行
```

**记忆来源优先级：**

| 优先级 | 来源 | 说明 |
|--------|------|------|
| 1 | 工作流上下文 | 当前流程前置节点产出，强制注入 |
| 2 | 团队记忆库 | 经确认的历史产出，自动检索 |
| 3 | 快手 Docs 索引 | 初始化时爬取的文档，自动检索 |

---

### 4.4 记忆沉淀机制

```
用户点击「通过 →」

      ├── 产出物写入工作流上下文（立即生效）
      │
      └── 生成记忆条目（状态：pending review）

            ├── 标签：自动打（节点类型 / 项目名 / 关键词）
            └── 等待 Owner 或本人审核
                  ├── 确认 → active，进入团队/个人记忆库
                  └── 拒绝 → 丢弃
```

**记忆分级：**

| 级别 | 写入条件 | 检索范围 |
|------|---------|---------|
| 个人记忆 | 本人通过节点，自动生成 | 仅本人可用 |
| 团队记忆 | Owner 审核确认后 | 全团队可用 |
| 工作流上下文 | 节点通过即写入 | 当前工作流内 |

## 6. 非功能需求

### MVP 内的边界约束

| 维度 | 约束 |
|------|------|
| 并发 | 单工作流实例同一时间只有一个节点处于执行中 |
| 上下文长度 | 单节点注入上下文（记忆 + 前置产出）不超过 128K token，超出时按优先级截断 |
| 记忆检索 | 每次自动召回 Top-5，响应时间 < 1s |
| Docs 同步 | 增量同步延迟 < 10 分钟 |
| 协作实时性 | 批注通知延迟 < 5s（站内消息） |

### MVP 不解决的问题

- **节点并行执行** — MVP 节点严格串行，不支持分支或并行

## 7. 成功指标

### MVP 验证目标

MVP 通过的判定标准是验证三个核心假设：

| 假设 | 验证指标 | 通过阈值 |
|------|---------|---------|
| 节点链式流转顺畅 | 完整跑完6个节点产出 PRD 的完成率 | ≥ 70% |
| 记忆注入有效降低调整成本 | 有记忆注入 vs 无记忆注入，节点平均对话轮数对比 | 有注入 < 无注入 30% |
| 记忆沉淀形成正向飞轮 | 第2次使用时，用户主动保留的自动召回记忆比例 | ≥ 50% |

### 辅助观测指标

| 指标 | 说明 |
|------|------|
| 节点通过率 | 各节点首次执行后直接通过的比例，识别哪个节点体验最差 |
| 平均节点迭代轮数 | 每个节点平均执行几轮才通过，越低越好 |
| 记忆 pending review 确认率 | 沉淀的记忆中最终被 active 的比例，过低说明产出质量差 |
| 协作参与率 | 有评审人参与批注的工作流占比 |

### 不作为 MVP 判断依据

- DAU / 留存率（用户量太小，数据无统计意义）
- PRD 质量评分（主观指标，MVP 阶段难以量化）
