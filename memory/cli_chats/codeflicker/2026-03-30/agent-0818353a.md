Replace the single shared `boolean` state `isPreviewImage` with a `string | null` state called `previewingCardIndex`. When it's `null`, no preview is open; when it holds a card's `index` (or `loadUrl` as fallback), that specific card's `FileCardImagePreviewer` will receive `preview={true}`.

- **Use `card.index` (falling back to `loadUrl`) as the unique preview key** — this mirrors the `patchIndex` pattern in `FileCard` (`item?.index || JSON.stringify(item?.content)`) and is the most semantically correct unique identifier already available on every card object.

**Why This Approach:**

The root bug is that a single boolean is shared across all rendered cards — flipping it to `true` causes every `FileCardImagePreviewer` in the list to enter preview mode simultaneously. The `string | null` approach is the minimal, zero-dependency fix. It requires no new components, no new stores, and no changes outside `CardsList/index.tsx`. It directly mirrors the per-instance `isPreview` state in `FileCard`, keeping both components consistent.

### Trade-offs Considered

**Alternative Approach 1: Use an index `number | null` (array position)**
- Pros: Very simple, guaranteed unique within the list.
- Cons: Array indices are positional — if the `data` array ever re-orders or is filtered, the wrong preview could open momentarily during re-render. `card.index` (the semantic card ID) is semantically stable.
- Why not chosen: `card.index` is the correct stable ID already used across the codebase (in `setActiveCardIndex`, `patchIndex` in `FileCard`, etc.). Using the array offset would be an inconsistency.

**Alternative Approach 2: Extract a `CardListImageItem` sub-component with its own local `isPreview` state**
- Pros: Perfectly encapsulated — each row manages its own open/close state, no lifted state needed. Mirrors `FileCard`'s design most closely.
- Cons: Requires creating a new file/component, more refactoring surface area, and the `handleClickCard` logic must also move into the sub-component or be passed as a prop. The `onClickItem` re-enable is still required.
- Why not chosen: Overkill for this fix. The `string | null` lifted state achieves the same isolation with a 4-line change, all within one file.

**Alternative Approach 3: Keep `boolean` state but scope it per-item via a `Map<string, boolean>`**
- Pros: No structural change to JSX.
- Cons: A `Map` in React state requires immutable update patterns (spread/copy), is more verbose, and adds no real benefit over a simple `string | null` since only one preview can ever be open at a time.
- Why not chosen: Unnecessary complexity.

### Edge Cases & Error Handling

- **`card.index` is `undefined`**: Use the fallback chain `card?.index ?? card?.content?.loadUrl`. `loadUrl` is already used in the current `previewSrc` prop, so it's always available for `IMAGE_CARD` types. If somehow both are missing (malformed data), `previewingCardIndex` will be set to `null` or `undefined`, which means no preview opens — a safe no-op rather than a crash.
- **Multiple rapid clicks on different image cards**: The `string | null` model means setting a new `previewingCardIndex` automatically closes the previous preview (only one can equal the new value). No special guard needed.
- **`onClickItem` being re-enabled for non-image cards**: The `handleClickCard` switch-case already handles all other card types correctly and calls `toggleRightPanelOpen`. The `IMAGE_CARD` branch still short-circuits before reaching that call. Re-enabling the prop restores the intended behavior for all card types.
- **The inner `img onClick` inside `renderIcon` vs. the outer card `onClick`**: The inner `img` has `e.stopPropagation()`, so clicking the thumbnail image will not bubble up to the `CardItemThumbnail`'s `onClick`. Both code paths (`renderIcon` img click and the outer card click hitting `handleClickCard`) end up calling `setPreviewingCardIndex(cardIndex)` — idempotent, no issue.
- **`previewSrc` being `null` for IMAGE_CARD**: `IMAGE_CARD` content uses `loadUrl` (not a blob URL from `useFile`). Unlike `FileCard` which fetches a blob via `previewFile`, `CardsList` directly uses `loadUrl`. The `FileCardImagePreviewer` handles `previewSrc: string | null` by falling back to rendering `children` in the preview modal — this is safe for the current data shape.

- **`loadUrl` as a key is non-unique if two IMAGE_CARDs share the same URL**: If the same image is listed twice in the `data` array, both would have the same key and both previews would open simultaneously. This is an unlikely real-world scenario, but if needed in the future, a composite key like `` `${item?.card?.index}_${item?.messageId}` `` can be used. For now, `card.index` should be unique by contract.
- **`onClickItem` re-enablement may reveal pre-existing bugs for non-IMAGE card types in `CardsList`**: Since the prop was commented out, other card type behaviors (PRODUCT_CARD, ARTIFACT_CARD, etc.) haven't been exercised through this path.
