### 问题真实性判断

**问题1：质量不可控** — **真问题，但归因有偏差**

AI 产出质量不稳定是真实痛点，但根因不是"缺乏结构化流程"，而是 **prompt 质量和上下文差**。结构化流程能部分缓解，但不能根治。PRD 用工作流来解决 prompt 工程问题，逻辑链条不够直接。

**问题2：经验无法传承** — **真问题，竞争最强的点**

这个问题在有一定规模的 PM 团队里确实普遍存在，且现有工具（Notion/Confluence）只能做静态文档，无法在产出过程中动态注入。结合内部 Docs 的冷启动设计也比较务实。这是这个 PRD 里**最有说服力的问题**。

**问题3：协作缺乏结构** — **弱问题，且方案与问题不匹配**

"PRD 评审依赖即时沟通"是协作工具的通用痛点，飞书/Confluence 批注功能已部分解决。PRD 设计的"批注 → 节点消化"机制解决的其实是"批注如何被 AI 执行"，而不是"协作结构化"。问题描述和方案描述指向了不同的东西。

---

### 其他关键问题

**最大风险：强假设目标用户没验证**

整个产品假设 PM 愿意把工作流程锁定在一个平台里跑完 6 个节点。但现实中 PM 的工作是高度非线性的——需求可能中途变，节点顺序会打乱。PRD 第 6 节明确"节点通过后不可撤销、不支持回退"，这在真实工作场景里会是严重摩擦点。

**记忆沉淀飞轮的前提条件未被验证**

第 7 节的核心验证假设之一是"记忆沉淀形成正向飞轮"，但这依赖 pending review 确认率足够高。而确认率高的前提是产出质量好，产出质量好的前提是记忆已经足够多——这是个先有鸡还是先有蛋的问题，冷启动靠 Docs 初始化是一个合理的解法，但 PRD 没有充分论证这个数据量是否真的够用。

**"内部 OpenAPI 体系"作为核心优势被过度强调**

PRD 多次提到"无需额外集成成本"，但 MVP 阶段只接了 Docs 一个 MCP，这个优势在 MVP 阶段对用户完全不可感知，说服力弱。

[2026-03-31 17:26:03] [assistant]: 回退能解决"流程摩擦"问题，但会引入新的复杂度：

**解决了什么**
- PM 不再因为"走错了就回不了头"而抵触使用，降低上手门槛

**带来的新问题**

1. **记忆污染**：如果节点 3 回退到节点 1 重跑，节点 2/3 产出的内容要不要写入记忆库？写了是噪音，不写需要有明确的"提交"动作——这会让产品交互变复杂。

2. **协作状态混乱**：多人场景下，A 在节点 4 做了批注，B 回退到节点 2 重跑，A 的批注悬空了——需要一套版本/分支机制，复杂度骤升。

3. **"流程价值"被稀释**：如果可以随意回退，流程的约束感消失，用户可能退化成"把它当成一个 AI 对话框"，结构化流程的差异化优势就没了。

**更务实的方案**：不做完整回退，而是支持**节点内重新生成**（同一节点多跑几次选最好的），以及**在后续节点里显式引用并覆盖前序结论**。这样既保留了流程结构，又给了 PM 纠错空间，记忆沉淀的边界也清晰。

核心判断：回退是个正确的方向，但要控制范围，否则会把一个 workflow 产品做成一个 chat with history 的产品。
