- 命名




- 名称单词要常见


- eg：接口定义前不必带 I（interface） 前缀


- 类名使用名词或名词短语命名（规范）


- 方法名使用动词或动词短语命名（规范）


- eg：区分 add 和 append、insert 命名，尽量更具体


- 使用一以贯之的前后缀（规范类）




- 函数




- 短小

- 只做一件事（函数名的抽象层次下定义的一件事）




- 如何判断函数是否不止做了一件事



- 能否再拆出一个函数（不仅只是单纯地重新诠释其实现）






- 函数的缩进层级不该多于两层


- 名称




- 别害怕长名称（长而具有描述性的名称）


- 包含动词和关键词




- 参数




- 参数越少越好，其不应超过 2 个参数

- 不使用标识参数



- 函数功能



- 做什么事

- 或 回答什么事




- 统一异常处理



- 使用 try catch() 代替 错误处理






- 注释



- 非必要不书写注释

- 必要注释



- 法律信息

- 提供信息的注释


- 对意图的解释


- 阐释


- 警示


- TODO 注释


- 放大某种看来不合理之物的重要性




- 应删除的注释



- 函数头


- 日志式注释

- 注释掉的代码






- 格式




- 垂直格式




- 减少文件的代码行数

- 不同功能的代码区域使用空白行隔开

- 变量神明应尽可能靠近其使用位置


- 相关函数仅可能放在一起，调用函数在被调用函数上方



- 水平格式



- 代码行宽度要尽可能小于 80 字符，上限为 120 字符

- 可借助空格区分一行代码的相关性


- 保证缩进


- 必须加 {}





- 对象和数据结构



- 对象暴露行为，隐藏数据。便于添加新对象类型而无需修改既有行为，但难以在既有对象中添加新行为


- 数据结构暴露数据，没有明显动作。便于向既有数据结构添加新行为，但难以向既有函数添加新数据结构



- 错误处理




- 使用抛出异常而非返回错误码




- 异常包含发生的环境说明




- 使用 try catch finally


- 函数的出入参禁止为 null




- 对于出参：尝试抛出错误或返回特例对象（非常规的返回值，非 null 或 undefined）


- 对于入参：需要做非空判断及抛出异常






- 类（类比组件）




- 系统应该由许多短小的组件而不是少量巨大的组件组成。每个小组件封装一个权责，只有一个修改的原因，并与少数其他类一起协同达成期望的系统行为。


- 为系统新增功能时，多看看是否有类似功能，尽量使用其封装后的组件/方法。




- 系统



- “一开始就做对系统”纯属神话

- 我们应该只去实现今天的用户故事，然后重构，明天再扩展系统、实现新的用户故事。




- 迭代




- 简单设计的四条规则



- 运行所有测试

- 不可重复

- 表达了程序员的意图

- 尽可能减少类和方法的数量






- 并发编程




- 并发防御原则




- 单一权责原则



- 分离并发相关代码与其他代码




- 推论：




- 限制数据作用域


- 使用数据副本

- 线程应尽可能地独立








- 逐步改进



- 要编写整洁代码，必须先写肮脏代码，然后再清理它。


- 测试驱动开发保证代码改进时不破坏代码功能；

- 优秀的软件设计，大都关乎分隔—创建合适的空间放置不同种类的代码。对关注面的分隔让代码更易于理解和维护。


- 进度可以重订，需求可以重新定义，团队动态可以修正。不能放任糟糕的代码一致腐败发酵
