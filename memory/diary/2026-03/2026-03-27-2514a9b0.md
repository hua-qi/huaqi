<div>### RAG

</div>
<div>
</div>
<div>通用的基础大模型无法满足实际业务需求的几方面原因</div>
<div>
</div>


- 知识的局限性



- 主流大模型的训练集基本都是构建于网络公开的数据，缺少实时性的、非公开的或私域数据



- 幻觉问题



- 所有深度学习模型的底层原理都是基于数学概率，模型输出实质上是一系列数值运算

- 对于模型自身不具备的知识或不擅长的任务场景，模型经常会一本正经的胡说八道



- 数据安全性



- 基于数据安全的考虑，公司不会将私域数据上传到第三方平台进行训练或推理，导致完全依赖通用大模型自身能力的应用方案不得不在数据安全和效果方面取舍





<div>
</div>
<div>RAG 是什么</div>
<div>
</div>
<div>Retrieval- Augmented Generation，检索增强生成</div>
<div>
</div>
<div>让大模型在回答问题时能够引用实时、可靠的外部知识，从而减少幻觉、提升准确性和时效性</div>
<div>
</div>
<div>RAG = 检索技术 + LLM 提示</div>
<div>
</div>
<div>基本流程</div>
<div>
</div>


- 问题理解

- 检索阶段



- 系统先去知识库、文档库甚至搜索引擎中，找到跟问题相关的内容



- 增强输入



- 将上述的相关内容和用户的问题一起打包发给大模型



- 生成答案



- 大模型基于检索到的资料，用自然语言生成调理清晰的回答





<div>
</div>
<div>RAG 的实现过程</div>
<div>
</div>


- 检索阶段



- 知识库构建

- 文本向量化

- 向量存储

- 查询检索



- 生成阶段



- 构造提示



- 将用户的问题转化为向量，在向量数据库中去找范围内的内容



- 相似度计算算法



- 点积（Dot Product / Inner Product）



- 两个向量的内积，数值越大代表约相似



- 欧氏距离（Euclidean Distance / L2 Distance）



- 两个向量之间直线距离，约近约相似



- 曼哈顿距离（Manhattan Distance / L1 Distance）



- 按照坐标轴方向计算距离









- 生成回答

<div>### 生成优化与推理加速技术

</div>
<div>
</div>
<div>生成优化与推理加速技术旨在解决 AI 生成模型的实时性差、内容质量不稳定、推理效率低等问题。</div>
<div>
</div>


- 流式输出技术



- 特点



- 边生成边返回



- 原理



- 网络传输



- 采用“分块传输”。生成的内容拆成多个数据块，实时传送给用户



- 计算调度



- “增量计算”拆分生成任务，完成一个子任务就输出，同时开始下一个子任务，实现生成与输出并行







- 解码策略



- 原因



- 解码策略决定从模型概率分布中选下一个 token 的方式，影响内容质量与多样性



- 场景应用



- 文本摘要



- 需准确连贯，常用束搜索（束宽 3～5）



- 对话生成



- 需自然多样，常用 Top-k（K = 10） 或 Top-p（p = 0.92）





- 常见解码策略



- 贪婪搜索



- 优势



- 计算效率高且易于实现



- 局限性



- 可能会导致文本重复且单调，因为它总是选择最有可能的单词，常常会使文本缺乏多样性





- 束搜索



- 原理



- 每次保留 Top-N 个候选序列（N 为束宽），基于这些序列继续生成，平衡质量与多样性，计算量随束宽增加而上升



- 初始化



- 以最有可能的初始标记作为候选序列开始



- 迭代扩展



- 对于每个候选序列，添加概率最高的前（k） 个下一个标记，创建新的序列



- 选择



- 根据累计概率为每个新序列打分，仅保留得分最高的前（k）个序列



- 终止



- 重复这个过程，直到达到最大标记长度，或者所有候选序列都以序列结束标记结尾







- 优势



- 与贪婪解码相比，通常能产生更连贯、更多样的问题



- 局限性



- 仍然存在重复问题，可能无法捕捉到所有可能输出的多样性。

- 计算成本可能很高，尤其当（k）值较大时





- Top-k 采样



- 原理



- 在每一步（t），模型从概率分布中随机采样，但只限于概率最高的前（k）个单词



- 优势



- 为生成的文本引入了随机性和多样性



- 局限性



- 与束搜索相比，有时可能会生成连贯性较差的文本



- （k） 值过小可能无法引入足够的多样性

- （k）值过大可能引入过多的随机性







- Top-p 采样（核采样）



- 原理



- 在每一步中，模型从概率分布中采样，只考虑那些累计概率至少达到（P）的单词



- 优势



- 平衡多样性和连贯性



- 通过关注最有可能出现的单词，同时允许一定的可变性，确保生成的文本既多样又连贯



- 动态调整



- 单词数量时动态调整的，产生更自然、更像人类生成的文本



- 控制创造性和连贯性



- 通过调整（p）的值，模型可以控制创造性和连贯性之间的权衡











- 长度控制技术



- 常见控制方法



- EOS Token 机制



- 原理



- EOS（End of Sentence）Token 是模型预训练阶段定义的“文本结束标记”，其作用是告知模型“当前文本已生成完毕”



- 优势与不足



- 优势



- 无需人工干预，模型可根据文本语义自动判断结束时机，生成内容的“语义完整性”更强



- 不足



- 当模型对语义理解不准确时，可能过早/过晚生成 EOS Token





- 优化策略



- 在微调阶段加入“长度监督数据“（如标注不同场景下的合理文本长度），让模型学习不同任务的 EOS Token 生成时机

- 设置”最小长度阈值“，避免模型过早停止生成





- 显式长度控制】



- 原理



- 通过直接设定“生成文本的长度范围”，强制模型在指定范围内生成内容



- 技术实现



- 输入提示约束



- 在用户查询中加入长度指令



- 参数配置约束



- 在推理接口中设置长度参数





- 优势与不足



- 优势



- 长度控制精度高



- 不足



- 当强制设定的长度与文本语义不匹配时，可能导致内容质量下降（文本内容冗余或被截断）





- 平衡策略



- “显式长度 + 语义匹配”，先通过显式参数设定长度范围，再通过语义检测算法判断生成内容是否完整。







- 平衡长度与质量



- 根据任务指定合理长度范围，结合解码策略调整，如显式控制 + Top-k/Top-p 避免冗余

- 引入质量评估机制优化





- 重复抑制算法



- 问题



- 模型在生成文本时，容易出现“词重复”或“短语重复”的问题



- 相关算法



- 惩罚因子



- 原理



- 通过对“已生成的 token”施加概率惩罚，降低其再次被选中的可能性。



- 优势与不足



- 优势



- 实现简单，计算开销低，能有效抑制词级别的重复



- 不足



- 对短语级别的重复抑制效果有限，

- 当惩罚因子过小时，肯呢个导致模型无法生成必要的重复词







- N-gram 重复检测



- 原理



- N-gram 指文本中连续的 n 个 token 组成的短语。n-gram 重复检测通过维护一个“已生成 n-gram 列表”，在每一步生成新 token 后，检查新形成的 n-gram 是否已存在列表中，若存在则使用以下两种方式抑制



- 概率抑制



- 降低当前生成 token 的概率，避免后续继续生成重复短语



- 强制替换



- 若重复次数超过阈值，则强制替换当前 token 为其他候选 token







- 优势与不足



- 优势



- 能有效抑制短语级别的重复，生成内容的流畅度更高



- 不足



- N 值的选择对效果影响较大，且可能误判“必要重复“





-  优化策略



- 采用“动态 n 值”机制，根据生成文本的长度调整 n 值（短文本用 n = 2，长文本用 n = 3）

- 建立“允许重复的术语库”，对知识库中的核心术语不进行重复抑制

<div>	2. 选择加速芯片</div>






- 模型框架与规模



- 模型基于 PyTorch 框架



- 优先选 GPU



- 基于 TensorFlow 且规模较大（100B 参数以上）



- 可考虑 TPU





- 部署环境



- 私有部署



- 优先选择 GPU



- 云部署



- 根据云厂商的资源情况选择（AWS 提供 GPU 实例，Google Cloud 提供 TPU 实例）





- 成本与能效



- 功耗敏感（边缘部署）



- 选择低功耗 GPU



- 大规模云服务



- 通过比较 GPU 与 TPU 的“每 token 推理成本”来选择更经济高效的方案







- 实际部署



- 性能需求



- GPU（单个任务的延迟较低



- 实时性要求较高的应用



- TPU（高吞吐量



- 批量处理任务





- 开发与维护成本



- GPU



- 成熟的开发工具链和广泛的社区支持



- TPU



- 开发环境相对封闭





- 可扩展性



- TPU



- TPU 提供了更好的可扩展性和集群管理能力





- 兼容性



- 确保所选硬件与现有软件栈和工具链兼容



- 长期规划



- 考虑技术的快速发展，选择硬件时应考虑其在未来几年内的可持续性和升级路径







- 缓存与预取技术



- 作用



- 优化数据访问，减少传输延迟，提升计算单元利用率



- 核心原理与应用



- 缓存技术



- 计算单元级缓存

- 模型参数缓存

- 请求级缓存



- 预取技术



- 基于规则预取

- 基于历史预取

- 基于硬件预取





- 优势与注意事项



- 优势



- 提升推理速度

- 降低硬件成本

- 支持大规模并发



- 注意事项



- 平衡缓存容量与命中率

- 平衡预取精度与资源开销

- 分布式场景确保缓存一致性







- 模型推理优化算法



- 模型推理优化算法通过对 Transformer 架构的计算流程、内存管理、服务调度进行深度优化，进一步提升推理性能

- 主流优化方案



- FasterTransformer



- 计算并行化



- 多头注意力并行计算，融合矩阵运算与激活函数，减少传输延迟



- 内存访问



- 调整数据布局适配 GPU，提升 GPU 利用率





- Trition



- 动态批处理



- 合并小批次请求为大批量，提升 GPU 利用率



- 模型实例管理



- 部署多实例并行处理请求，支持模型/流水线并行，适配大模型



- 多框架兼容



- 支持多框架模型，自动应用优化策略
